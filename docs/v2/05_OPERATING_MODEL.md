# 운영 모델 — 사람×AI 페어 3트랙

> 상태: **현재 권장 운영 기준.** 상위 문서: [README.md](./README.md).
> 전제: 기획·디자인·개발 세 트랙을 **각각 사람 1명이 자기 AI를 지휘해** 수행한다. 개발 과정의 생성형 AI는 사람이 지휘하고, 제품의 contextual ML은 승인된 feature·model·calibration 계약 안에서만 예측한다. 최종 배포와 대외 주장은 사람이 판단하고 책임진다.
> 이전 두 운영안과의 관계 — [`PLAN.md`](../PLAN.md)(사람만, 2일), [`PLAN_AI.md`](../PLAN_AI.md)(AI 자율 조직): 이 문서는 두 안의 중간에 해당하는 v2 권장안이다. AI 자율 조직의 에이전트 역할 분류는 유지하되, **모든 에이전트의 트리거가 "사람의 지시"로 바뀐다.**

---

## 0. 페어 체제의 3원칙

1. **AI는 계약 안에서 수행하고, 사람은 경계와 승격을 결정한다.** contextual ML은 승인된 입력으로 확률을 계산하지만 feature source 승인, 타깃·기준선·가드레일, 배포·명칭 강등은 사람의 것이다. 생성형 LLM은 런타임 등급·대안·행동을 결정하지 않는다.
2. **AI 산출물은 검증 전까지 가설이다.** "fixture 또는 테스트로 재현되지 않는 주장은 항상 가설"(README §6)은 AI가 만든 코드·문서·수치에 가장 강하게 적용된다. 실제로 초안 단계에서 AI가 존재하지 않는 데이터("보정 대수")를 전제로 화면을 설계한 사례가 있다([V1 정합성 검증](../research/V1_정합성검증.md) 치명1).
3. **페어 간 정렬은 대화가 아니라 계약으로 한다.** 사람끼리는 말로 합의해도 되지만, 그 합의는 반드시 `contracts/`에 반영된 뒤에야 각자의 AI에게 전달된다. AI 세 대가 서로 다른 구두 합의 기억으로 일하는 순간 드리프트가 시작된다.

## 1. 트랙별 분담 — 사람의 몫 / AI의 몫

### 1.1 기획 담당 × AI

| | 내용 |
|---|---|
| **사람의 몫 (결정·책임)** | 스코프 결정(무엇을 안 만들지), 게이트 판정(G0~G4 통과 여부 선언), 날씨·휴일·행사·공간 source의 제품 사용 승인, `contextual_ml` 명칭과 강등 판단, 모델 KPI·가드레일 승인, 현장 인터뷰 수행, 발표, 대외 커뮤니케이션 |
| **AI의 몫 (수행)** | 공공데이터·공지·사례 리서치와 사실 대장 갱신([확인]/[추정] 태그), context source 비교표와 data card 초안, 심사기준 역산 문서·발표 덱 초안, slice·ablation KPI 해석 리포트, 인터뷰 질문지 설계·기록 정리 |
| **주요 지시 대상 산출물** | `02_DATA_AND_EVALUATION.md`의 사실 대장, 발표 자료, 완료기준 측정 기록 |

**지시할 때 반드시 거는 제약** (지시문에 상주):
- 금지 표현 5종(README §1 — "고장 자동 탐지", "보정된 대수" 등)을 산출물에 쓰지 않게 한다.
- 모든 수치 주장에 출처([확인]/[추정] + 근거 문서)를 달게 한다 — 출처 없는 수치는 반려.
- “날씨를 고려한다” 같은 모델 주장은 B2 기준선 대비 증분 성능과 해당 feature block ablation이 없으면 반려한다.
- 기능 제안이 오면 "이건 P0/P1/P2 어디인가, 어떤 게이트 뒤에 있나"를 먼저 답하게 한다 — 스코프 폭발 차단.

### 1.2 디자인 담당 × AI

| | 내용 |
|---|---|
| **사람의 몫** | 미적 방향 선택(AI가 낸 시안 중 픽), 실사용자 사용성 테스트 수행·관찰, "이 화면이 오해를 낳는가"의 최종 판단 |
| **AI의 몫** | 컴포넌트·화면 코드 생성(코드로 디자인), `화면 × response_mode × freshness × context_status × prediction_basis × 라이트·다크` 매트릭스 전수 생성, context evidence의 출처·시각·강등 상태 표현, 접근성 자동 점검·수정, 시각 회귀 테스트 |
| **주요 지시 대상 산출물** | 프론트 컴포넌트, `contracts/enums.json`의 UI 라벨 반영, 상태별 스크린샷 세트 |

**지시할 때 반드시 거는 제약**:
- 세션 시작 시 `contracts/enums.json` + `contracts/invariants.md`를 읽힌다. **계약에 없는 필드·상태·라벨을 발명하면 반려** ([V1 정합성 검증](../research/V1_정합성검증.md) 치명1의 재발 방지 — 이 위반은 사람이 눈으로 잡기 어렵고, 계약 대조로만 잡힌다).
- 관측과 추론의 분리(README §4-2): 원본 수량·기준 시각·예측 등급·커뮤니티 신고·공식 상태가 한 컴포넌트에 섞이면 반려.
- context evidence는 최대 2개, 출처와 시각을 포함하고 “비 때문에 소진” 같은 인과 문구나 raw SHAP 수치를 만들지 않게 한다.
- `context_status=partial|unavailable`과 `inventory_temporal|current_stock` 폴백을 정상 상태만큼 완성도 있게 설계하게 한다.
- `response_mode=demo`이면 공식 재고가 live source여도 화면 전체의 실제 길찾기·신고 CTA와 KPI 전송을 차단하게 한다.
- 대표 화면만 만들고 끝내지 못하게 한다 — 상태 매트릭스 전수를 명시적으로 요구(사람 디자이너가 빠뜨리는 지점을 AI의 강점으로 메우는 것이 이 페어의 존재 이유).

### 1.3 개발 담당 × AI

| | 내용 |
|---|---|
| **사람의 몫** | API 키 발급·보관(시크릿은 AI에게 주지 않는다), station/context fixture 충분성 판단, feature contract·타깃·기준선 승인안 작성, model card 소유, 배포·머지 승인, 비용 상한 |
| **AI의 몫** | station·weather·calendar·event·graph adapter, versioned context snapshot, point-in-time feature builder, B0/B1/B2 재고 기준선·B3 투명한 맥락 기준선과 contextual 후보·calibration, prediction log/API 서버 구현, OpenAPI·누수·schema·fallback 테스트, rolling-origin·공간/행사 holdout·ablation 평가 |
| **주요 지시 대상 산출물** | station/context fixture, `StationObservation`·`ContextSnapshot`·`FeatureVector` 계약, model/data card, feature lineage, 계약·누수·fallback 테스트, Idempotency·익명 세션 처리 |

**지시할 때 반드시 거는 제약**:
- `contracts/openapi.yaml`이 원본 — 코드에서 스키마를 임의 확장하면 반려. 스키마 변경이 필요하면 "계약 PR 먼저"를 요구하게 한다.
- **G0 미통과 상태에서 live 경로 구현 금지** — fixture 없이 "아마 이런 응답일 것"으로 짠 코드는 전부 폐기 대상임을 지시문에 명시.
- 모든 외부 context 행의 `event_time/available_at/ingested_at/source_version`을 보존하고 `available_at <= prediction_time` point-in-time join만 허용한다. 사후 실제 날씨·관객 수·최종 취소 상태는 학습 입력에서 반려한다.
- context 결측을 `비 없음`·`행사 없음`으로 채우지 않는다. 별도 검증된 `contextual_ml → inventory_temporal → current_stock` 경로와 각 model/calibration/threshold 버전을 테스트한다.
- 사용자 위치, 커뮤니티 신고, capability token, 미래 대여 이벤트를 모델 feature로 넣지 않는다. 위치는 요청 순간의 거리 정책에만 사용하고 학습·예측 로그에 보존하지 않는다.
- 완료 보고는 반드시 테스트 실행 로그 첨부 — "구현했습니다"는 완료가 아니고, 통과 로그가 완료다.
- `x_pos/y_pos`의 실제 좌표 의미와 CSV 인코딩을 fixture·원본에서 탐지하게 하고 추정값을 하드코딩하지 않는다. `is_missing`과 유효한 0대도 분리하도록 세션 시작 컨텍스트에 포함한다.

### 1.4 데이터·모델 교차 승인

3인 체제에서 별도 직군을 추가하지 않고 다음 책임을 명시한다.

| 책임 | 주 소유자 | 독립 승인자 | 승인 증거 |
|---|---|---|---|
| source·feature lineage, point-in-time join, train-serving parity | 개발 담당 | 기획 담당 | data card, 누수 테스트, feature 재현 로그 |
| 타깃·기준선·slice·ablation·calibration | 개발 담당 | 기획 담당 | model card, rolling/holdout 평가, 임계값 버전 |
| 사용자 문구·context evidence·오해 가능성 | 기획 담당 | 디자인 담당 | 문구표, 사용성 결과, 사실 일치 검사 |
| shadow→canary→공개 승격 | 기획 담당 | 개발·디자인 공동 | 가드레일 대시보드, rollback 훈련, 서명된 gate 기록 |

모델을 만든 개발 담당이 혼자 승격을 승인할 수 없고, 기획 담당도 평가 코드를 실행한 재현 로그 없이 성능 주장을 승인할 수 없다.

## 2. 공통 지시 프로토콜 (세 사람 모두)

### 2.1 세션 시작 — 컨텍스트 앵커

AI에게 일을 시키기 전에 매 세션 다음을 읽힌다 (붙여넣기 템플릿):

```
다음 문서를 먼저 읽어라. 이것이 이 프로젝트의 현재 기준이다:
1. docs/v2/README.md          (전체 기준, 금지 표현, 권위 우선순위)
2. docs/v2/contracts/         (openapi.yaml, enums.json, invariants.md — 기계 판독 계약)
3. docs/v2/0N_*.md            (내 트랙 문서: 기획=01·02 / 디자인=01 / 개발=02·03·04)
규칙:
- 계약에 없는 필드·상태·표현을 만들지 마라. 필요하면 "계약 변경 제안"으로 따로 보고하라.
- 모든 사실 주장에 [확인]/[추정] 태그와 근거를 달아라.
- 확인되지 않은 것은 확인되지 않았다고 말하라. 추정을 확정처럼 쓰면 반려한다.
- README §1의 금지 표현 5종을 어떤 산출물에도 쓰지 마라.
- contextual 예측 작업은 feature source의 available_at, point-in-time join, 기준선 B0/B1/B2/B3, ablation, calibration과 fallback 증거를 함께 제출하라.
- 모델 중요도나 context evidence를 인과관계로 표현하지 마라.
```

### 2.2 산출물 수령 — 3단계 검증 (사람이 직접)

| 단계 | 확인 | 걸리는 시간 |
|---|---|---|
| ① 계약 대조 | 산출물의 필드·enum·표현이 `contracts/`에 있는 것만 쓰는가 | 1~2분 |
| ② lineage·누수 | feature가 예측 시점에 실제 이용 가능했고 학습·서빙 변환이 같은가 | 자동검사 + 표본검사 |
| ③ 근거 확인 | 수치·사실 주장마다 출처/fixture/기준선·ablation·테스트 로그가 붙어 있는가 | 2~3분 |
| ④ 교차 검증 | 중요 산출물(계약, Core 모델, 발표 수치)은 **옆 트랙 사람에게 넘겨 그 사람의 AI로 반박**시킨다 | 필요 시 |

④가 이 체제의 핵심 장치다: 초안 단계의 [적대적 검증](../research/V1_정합성검증.md)이 트랙 간 모순 10건을 잡았던 패턴을, "다른 사람이 지휘하는 다른 AI"로 상시화한 것이다. 자기 AI에게 자기 산출물을 검증시키면 같은 맹점을 공유하므로 반드시 **다른 페어**가 반박한다.

### 2.3 지시 기록

재현이 필요한 지시(계약 변경, Core 산출 규칙, 평가 기준)는 채팅에 흘리지 말고 [`prompts/`](./prompts/)에 `<트랙>_<주제>.md`로 남긴다. AI가 바뀌거나 세션이 끊겨도 같은 지시로 같은 산출물을 재현할 수 있어야 한다.

## 3. 페어 간 연결 — 계약이 회의를 대신한다

### 3.1 연결 구조

```
[기획 사람] ──지시──► [기획 AI]
     │ 합의(구두·즉석)                    산출물
[디자인 사람] ──지시──► [디자인 AI]      ↓
     │                          ┌──────────────────┐
[개발 사람] ──지시──► [개발 AI]  │  docs/v2/contracts/ │ ← 유일한 공식 합의 채널
                                │  openapi / enums /  │
                                │  invariants         │
                                └──────────────────┘
                                 세 AI가 모두 이것만 바라본다
```

- **사람 간 합의 → 계약 반영 → AI 지시** 순서를 지킨다. 계약에 안 넣고 구두 합의로 각자 AI에게 지시하는 순간, 세 AI의 세계관이 갈라진다.
- 계약 변경 절차는 README §6을 따른다: 제안(사용자 가치 + 완료 기준) → `contracts/` 먼저 수정 → 영향 문서 동시 갱신 → 상대 트랙 사람이 자기 AI로 영향 검토(§2.2-③) → 반영.
- 프론트·백엔드·mock은 계약에서 생성/검증하므로, 계약이 맞으면 페어끼리 코드 수준 조율은 거의 필요 없다.

### 3.2 사람이 모이는 시점 (AI 없이)

| 시점 | 안건 | 판정자 |
|---|---|---|
| 계약 동결 | openapi·enums·문구 확정, 이후 변경은 "nullable + 기본 숨김"만 | 3인 합의 |
| 게이트 판정 | G0~G4 증거 검토 — fixture·매칭률·평가 결과를 보고 통과/강등/차단 선언 | 기획 주관 |
| 모델 승격 | data/model card, B0~B3 대비 성능, slice·ablation, calibration, shadow 가드레일, rollback 검토 | 기획 승인 + 개발 책임 |
| E2E 통합 | mock → 실제 API 교체, 데모 시나리오 통주 | 개발 주관 |
| 완료기준 검수 | `01_PRODUCT_MVP.md` KPI 수치 실측 | 기획 주관 |
| 사용성·접근성 검수 | 실사용자 테스트, 검수 항목(로드맵 단계 E) | 디자인 주관 |

게이트 판정만은 AI에게 위임 금지 — 게이트는 "증거가 충분한가"라는 판단이고, 증거를 만든 AI가 증거의 충분성을 판정하면 순환이다.

### 3.3 로드맵 단계별 주도 페어 (04_EXECUTION_ROADMAP 매핑)

| 단계 | 주도 | 지원 |
|---|---|---|
| A 데이터 게이트 | 개발 페어 (station/context fixture·lineage·프로파일) | 기획 페어 (source 사용 승인·사실 대장) |
| B 계약과 골격 | 개발 페어 | 전원 (계약 동결) |
| C Core | 개발 페어 (feature builder·기준선·contextual model·calibration·fallback) | 기획 페어 (승격·문구 판정), 디자인 페어 (context/fallback 상태 렌더링) |
| D 현장 신고 | 디자인 페어 (플로우) + 개발 페어 (Idempotency·토큰) | 기획 페어 (안전결함 안내 문구) |
| E 데모·검수 | 기획 페어 (검수·발표) | 디자인 페어 (접근성), 개발 페어 (오프라인 완주) |
| S0~S4 (이후) | 단계 성격에 따라 동일 원칙 — 수집·평가는 개발, 섀도 판정·공개 결정은 기획, 노출 설계는 디자인 | — |

## 4. 이 체제 특유의 리스크

| 리스크 | 증상 | 대응 |
|---|---|---|
| **사람이 AI를 안 믿거나 다 믿음** | 전부 재검토(속도 상실) 또는 무검증 수용(V1 치명1 재발) | §2.2의 3단계 검증을 고정 루틴으로 — 1~5분짜리 얕은 검증을 항상, 깊은 검증은 교차로 |
| **지시 드리프트** | 같은 개념을 세 사람이 다른 말로 지시 → AI 산출물 어긋남 | 용어는 `enums.json`의 코드값·라벨만 사용. 지시문에도 계약 용어를 쓴다 |
| **AI 과잉 생산** | 시키지 않은 기능·문서·추상화가 딸려옴 | 지시에 산출물 목록을 명시하고 "목록 외 산출물은 만들지 마라" 조항. 수령 시 목록 외 산출물은 검토 없이 폐기 |
| **컨텍스트 누락 지시** | 앵커 없이 시켜서 AI가 구버전 문서(PLAN 등) 기준으로 작업 | §2.1 템플릿 강제. 구 문서에는 이미 "이전안" 헤더가 붙어 있으므로 AI가 읽어도 현행 기준을 우선하게 지시 |
| **책임 공백** | "AI가 그렇게 만들었다"로 원인 회귀 | 모든 머지·게이트·발표 수치는 사람 이름으로 승인 — AI는 책임 주체가 아니다 |
| **AI 이름만 붙인 기준선** | 현재 재고·시간 임계값인데 contextual AI로 발표 | B2보다 증분 개선과 feature-block ablation이 없으면 `contextual_ml` 명칭 금지 |
| **시점 누수·train-serving skew** | 사후 날씨/행사 값으로 오프라인 성능만 높음 | `available_at <= prediction_time`, fold 경계 purge, online/offline feature parity 계약 테스트 |
| **context 결측 은폐** | feed 장애가 `비 없음`·`행사 없음`으로 입력됨 | missing을 별도 상태로 유지하고 검증된 basis로만 강등, fallback률 경보 |
