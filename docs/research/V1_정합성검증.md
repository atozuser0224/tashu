# V1 정합성 검증 기록

> **문서 상태: 역사적 감사 기록.** 지적사항의 현재 폐쇄 기준은 [`docs/v2/README.md`](../v2/README.md)와 v2 계약이다.

## 치명(반드시 수정)

**1. 디자인의 "보정된 실제 이용가능 대수" 표시가 기획의 원칙을 정면으로 위반하고, 존재하지 않는 데이터를 전제로 한다.**
- 위치: 디자인 2-1(지도 뷰) "마커 배지 숫자 = '보정된 실제 이용가능 대수': API `parking_count` 원본값에서, 시스템이 고장의심(AI)으로 판정한 대수를 차감해 표시" / 2-2(정류장 상세) "보정된 이용가능 0대"
- 문제: 기획 1.4.1은 "절대 '실제 이용가능 대수는 X대'라고 새로운 숫자를 확정 표기하지 않는다"고 명시적으로 못박았는데, 디자인은 정확히 그 금지된 행위(신규 숫자 계산·표기)를 하고 있다. 더 심각한 건 그 계산 근거인 "고장의심으로 판정한 대수"가 리서치 어디에도 없다는 점이다 — R3의 이상탐지 축A/축B는 정류장 단위 "의심 스코어"(0~1 신뢰도)만 산출하며, "이 정류장 N대 중 몇 대가 고장으로 추정된다"는 자전거 단위 카운트는 R1이 명시적으로 "존재하지 않는다"고 확인한 개별 자전거 데이터가 있어야 계산 가능한 값이다. 즉 디자인이 리서치에 없는 필드(고장 추정 대수)를 지어냈다.
- 수정 지시: 디자인 2-1/2-2에서 "보정된 이용가능 대수" 문구와 숫자 차감 로직을 제거하고, 기획 1.4.1 원칙대로 "원본 `parking_count` 값 + 신뢰도 배지(낮음/중간/높음)"만 병기하는 구조로 되돌릴 것. 두 화면의 목업 텍스트("보정된 이용가능 0대" 등)도 함께 수정. 만약 팀이 정말 보정 숫자를 노출하고 싶다면, 그 전에 기획 1.4.1 자체를 개정하고 3부 데이터모델·파이프라인에 해당 산출값(및 그 산출식·오차범위)을 정의하는 작업이 선행되어야 하며, 디자인만 단독으로 앞서가면 안 된다.

**2. 로드맵상 "독립 웹·앱 공개" 시점이 기획과 개발에서 어긋나고, M4의 정의 자체가 두 섹션에서 다르다.**
- 위치: 기획 1.6 로드맵 M2행 "독립 웹·앱(형태①)으로 공개" vs 개발 8장 M2("체크인 API 최소 기능"만, 프론트엔드 언급 없음) 및 M4("이용자 PWA... 운영 대시보드... 정비사 체크인 앱" 구축)
- 문제: 기획은 M2(6~8주 시점)에 이용자가 쓸 수 있는 웹·앱이 공개된다고 약속하지만, 개발의 실제 구현 순서는 그 웹앱(PWA)을 M3 이후인 M4에서야 만든다. 게다가 기획의 M4("트랙레코드 기반 제휴 추진 + 정비 체크인 확장")와 개발의 M4("앱/대시보드 구축")는 같은 라벨 아래 완전히 다른 산출물을 가리켜, 두 로드맵을 나란히 보면 서로 대응이 안 된다.
- 수정 지시: (a) 개발 8장 M2에 "원본 데이터 그대로 보여주는 최소 웹뷰(신고 폼 포함)" 구축 항목을 추가해 기획 M2의 "독립 웹·앱 공개" 약속과 실제로 맞추거나, (b) 기획 1.6 M2행의 "독립 웹·앱으로 공개" 문구를 "체크인 API 공개(웹 UI는 M4)"로 낮춰 현실화할 것. 또한 두 로드맵의 M-번호 대응관계를 표로 명시하거나(예: 개발 M4 앱 구축 = 기획 M2 산출물의 지연 구현임을 각주로 표기), 라벨이 겹치지 않도록 개발 쪽 마일스톤명을 M2b/M4-app 등으로 분리할 것.

**3. 정비사 체크인의 세부 상태·배정(assign) 기능이 기획/디자인에는 있는데 개발 데이터모델·API에는 빠져 있다.**
- 위치: 기획 1.4.3(b) `flagged→assigned→arrived→inspecting→returned_to_service|withdrawn|parts_pending`, 디자인 2-4("오늘의 점검" 리스트, GPS 도착체크인), 디자인 2-5(대시보드 "[정비사 배정]" 버튼) vs 개발 3부 §3 DDL `checkin_status ENUM('pending','verified','dismissed','resolved')`, §6.1 API 표
- 문제: 개발의 `checkin_report.status`는 4단계뿐이라 "배정됨/도착/점검중"을 구분할 방법이 없고, `anomaly_event`에는 담당 정비사를 지정하는 필드가 아예 없다. 디자인 2-5가 그린 "[정비사 배정]" 버튼과 2-4의 "오늘의 점검 리스트" 화면 모두 이를 뒷받침할 엔드포인트(정비사 배정 API, 오늘의 점검 리스트 조회 API)가 §6.1에 존재하지 않는다. 즉 세 섹션이 그리는 정비 워크플로가 개발 단계에서 실제로 구현 불가능하다.
- 수정 지시: 개발 §3 DDL에 `anomaly_event.assigned_mechanic_id`, `assigned_at` 필드와, `checkin_report`에 `mechanic_status`(assigned/en_route/arrived/inspecting/completed) 세부 상태 컬럼을 추가할 것. §6.1 API 표에 `POST /api/v1/anomaly-events/{id}/assign`(정비사 배정)과 `GET /api/v1/mechanic/assignments`(오늘의 점검 리스트) 엔드포인트를 신설해 기획 1.4.3(b)·디자인 2-4/2-5와 대응시킬 것.

**4. 이용자용 오탐 피드백 버튼("정확한가요?")에 대응하는 API가 없다.**
- 위치: 디자인 4장/2-2 — "정류장 상세와 알림 카드 모두에 '이 판정이 정확한가요?' 버튼을 상시 노출" (정류장 상세는 이용자 공개 화면) vs 개발 6.1 API — 유일한 피드백 엔드포인트 `POST /api/v1/anomaly-events/{id}/feedback`가 "운영자 인증" 전용
- 문제: 디자인은 익명 이용자도 사용하는 공개 화면(정류장 상세)에 정탐/오탐 버튼을 넣었지만, 개발이 정의한 피드백 API는 운영자 인증이 필요해 익명 이용자가 호출할 수 없다. 이는 과제에서 요구한 "오탐 대응 루프"가 이용자 경로에서는 실제로 작동하지 않는다는 뜻이다.
- 수정 지시: 개발 6.1에 이용자용 별도 엔드포인트(예: `POST /api/v1/anomaly-events/{id}/user-feedback`, `device_id` 기반 익명 허용 + rate-limit)를 신설하거나 기존 엔드포인트를 "운영자 인증 또는 device_id 인증" 이원화하고, §3 DDL에 `operator_feedback`과 별도로 `user_feedback_count`/`user_feedback_agree_ratio` 등 신뢰도 가중치가 다른 컬럼을 추가할 것. 디자인 4장에도 "이용자 피드백은 운영자 피드백보다 낮은 가중치로 반영된다"는 문구를 추가해 두 피드백 소스의 신뢰도 차이를 명시할 것.

## 권고

**5. `anomaly_event.status`의 `pending_review`가 기획의 상태도에 없다.**
- 위치: 개발 §3 DDL `anomaly_status` enum, §6.2 전이표 vs 기획 1.4.3(c) `normal → ai_flagged → suspected/confirmed → resolved/auto-cleared`
- 수정 지시: 기획 1.4.3(c)의 전이도에 `ai_flagged → pending_review → suspected/confirmed` 형태로 중간 단계를 추가해 개발 스키마와 맞추거나, 개발 쪽에서 `pending_review`를 `ai_flagged`에 통합해 상태 수를 줄일 것 — 어느 쪽이든 세 섹션이 동일한 상태 이름과 전이 순서를 쓰도록 한 번 더 맞출 것.

**6. 개발 6.1의 "§0-5 참조" 인용이 틀렸다.**
- 위치: 개발 6.1 API 표, `POST /mechanic/checkins` 행 — "정비사 인증(운영기관 제휴 성사 시, §0-5 참조)"
- 문제: §0-5는 "API 토큰 수동승인·소요기간 비공개"에 관한 항목이지 정비사 인증/MOU와 무관하다.
- 수정 지시: 인용을 §0-5에서 기획 1.2.1(형태② 운영기관 제휴) 또는 §0 표에 항목을 신설(예: "8. 정비사 체크인은 운영기관 인력 접근이 있어야 성립, 근거: R2 §3-b")해 정정할 것.

**7. "전기자전거" 표현이 리서치보다 구체화되어 있다.**
- 위치: 기획 1.1.1 "전기자전거의 경우 태양광 충전 패널이 낙엽·쓰레기 등으로 가려져 방전된 사례가 확인됨"
- 문제: R2 원문은 "타슈 자전거의 배터리 고갈 원인으로 태양광 패널이 가려짐"이라고만 하며, 타슈 자전거가 전기자전거(PAS)인지는 R1~R4 어디에서도 명시적으로 확인되지 않았다(스마트 잠금·GPS용 배터리일 가능성도 있음).
- 수정 지시: "전기자전거의 경우"라는 한정을 제거하고 "타슈 자전거(잠금장치·GPS 구동용 배터리 탑재)의 배터리가 태양광 패널 가림으로 소진되는 사례" 정도로 R2 원문 수준에 맞춰 완화할 것.

**8. 기획 1.7의 비용 추정치가 개발 §5.6의 재계산 결과와 불일치한다.**
- 위치: 기획 1.7 리스크표 "현재 문서의 비용 추정(월 대략 수만~십수만 원대)은 300개소 가정 기준" vs 개발 §5.6 재계산 "1,000~1,500개소 기준 월 약 $240~720(약 32만~97만 원)"
- 문제: 기획 스스로 "규모 추정치 불일치"를 리스크로 지적해놓고, 정작 인용하는 비용 숫자는 (연구에서 실측 확인된 실제 규모인) 1,000~1,500개소가 아니라 R4의 300개소 가정치에 머물러 있다.
- 수정 지시: 기획 1.7의 해당 문구를 개발 §5.6의 재계산치("정류장 1,000~1,500개소 기준 월 약 $240~720, 약 32만~97만 원")로 교체할 것.

**9. 콜드스타트 기간의 이용자 화면 상태가 디자인에 명세돼 있지 않다.**
- 위치: 개발 §4.4(콜드스타트 2~4주 표) vs 디자인 2부 전체
- 문제: 개발은 1~2주차엔 "자동탐지 없음, 체크인만 가동"이라고 명시하는데, 디자인 2-1/2-2는 AI 배지·신뢰도 표시가 상시 존재하는 것을 전제로 목업을 그려 두 단계 사이의 화면 상태 차이를 다루지 않는다.
- 수정 지시: 디자인 2-1/2-2에 "AI 신호 없음/데이터 축적 중" 상태(배지 자체를 숨기거나 "분석 중" 라벨로 대체)를 상태값 표(2장 "상태" 항목)에 추가하고, 개발 §4.4의 주차별 표와 짝지어 어느 시점부터 AI 배지가 노출되는지 명시할 것.

**10. "정류소"/"정류장" 용어가 문서 전체에서 혼용된다.**
- 위치: 기획 1부 전반(두 용어 혼용) vs 디자인/개발(주로 "정류장")
- 수정 지시: 한 용어("정류장" 권장, 디자인·개발과 일치)로 기획 1부 전체를 일괄 치환할 것.
