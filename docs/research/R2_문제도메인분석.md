# 문제 도메인 분석: "고장·이용불가 자전거가 앱에는 이용가능으로 표기되는 문제"

## 0. 조사 베이스라인 (기지 사실)

- [확인] 타슈 오픈API `https://bikeapp.tashu.or.kr:50041/v1/openapi/station` 은 살아있으며, `api-token` 헤더 없이 호출 시 `{"error_code":"1001","message":"권한이 없습니다."}` (HTTP 400)를 반환한다. (사용자 제공 기지 사실)
- [확인] 타슈 앱 내 [자전거정보 > Open API] 메뉴에서 토큰 신청이 가능하며, 공지 문서(`bike.tashu.or.kr/noticeDetail.do?seq=28`)에 명시된 응답 필드는 `id, name, name_en, name_cn, x_pos, y_pos, address, parking_count` 로, **정류장별 "대여가능 자전거 수량"(총량)만 제공하고 고장 대수·거치대 상태를 구분하는 필드는 없다.** (출처 페이지를 AI 요약 도구로 읽은 결과이므로 실제 JSON 스키마와 완전히 동일한지는 재검증 필요 — 신뢰도 중상)

이 두 사실이 이번 분석 전체의 전제다: **우리가 접근 가능한 것은 정류장 단위 "합계" 뿐이고, 개별 자전거의 상태(정상/고장/배터리소진 등)는 API에 노출되지 않는다.**

## 1. 기존 고장 감지·처리 방식 조사

### 1-1. 타슈 (대전)
- [확인] 타슈는 2020~2022년 "타슈 시즌2"로 전면 개편, 자전거 자체에 GPS·스마트 잠금장치를 부착하고 QR코드로 대여/반납하는 방식을 도입했다 (구형 무인거치대의 잦은 고장·결제오류 개선 목적). 즉 자전거 단위 위치추적 자체는 인프라적으로 존재하나, 이 정보가 오픈API로 외부에 공개되진 않는다. (출처: 충청투데이, 경향신문, 굿모닝충청 보도)
- [확인] 고장/파손 신고는 앱 고객센터 또는 홈페이지 '고객지원', 전화(1899-2282)로 접수하며, QR코드 1회 스캔 또는 대여코드 기억 + **사진 필수** 조건이 있다. → 즉 현재도 "이용자 신고형" 체크인이 존재하지만 절차가 무겁고(사진 필수, 전화/홈페이지 경유) 앱 내 원탭 신고 UX는 아닌 것으로 보인다.
- [확인] 실사용 민원 사례(2차 출처, 블로그 heyhey.kr): 반납을 마쳤는데도 앱엔 "미반납"으로 표시되어 요금이 자동 청구되는 사례가 보고됨. 이는 본 과제(고장→이용가능 오표기)의 **거울상 문제**(정상 반납→미반납 오표기)로, 근본 원인이 동일함을 시사한다: **서버와 현장 단말기 간 실시간 동기화 지연/누락.**
- [확인, 2차 출처] 타슈 자전거의 배터리 고갈 원인으로 "바구니 하단 태양광 패널이 낙엽·쓰레기·짐 등으로 가려짐"이 지목됨 → 앱엔 이용가능으로 뜨지만 실제 대여 시도하면 전원이 안 들어오는 전형적 사례.

### 1-2. 따릉이 (서울)
- [확인] 앱 내 '대여 > 고장신고' 메뉴에서 QR코드 스캔 또는 자전거 번호+고장 위치 직접 입력으로 신고 가능. 고객센터(1599-0120) 전화 신고도 병행.
- [확인] 서울 열린데이터광장의 실시간 대여정보 API는 `parkingBikeTotCnt`(거치된 자전거 수), `rackTotCnt`(거치대 총수) 등 **정류장 합계 필드** 위주이며, 개별 자전거의 고장 여부 필드는 확인되지 않음 — 타슈와 동일한 구조적 한계.

### 1-3. 지방 공영자전거 공통 패턴 (2차 출처, 블로그 비교글)
울산 피프틴(앱 배터리 90%인데 실제 방전), 세종 어울링(앱 자전거 있음 표시인데 현장엔 없음), 속초/춘천 타조(앱은 간편하지만 실물 정비상태 부실) 등 조사된 모든 지역 공영자전거에서 **"서버-현장기기 실시간 동기화 부족"이 공통 근본원인**으로 지목됨. → 이는 대전 타슈만의 문제가 아니라 한국 공영자전거 시스템 전반의 구조적 문제이며, 우리 솔루션이 다른 지자체로 확장 가능한 도메인임을 시사한다.

### 1-4. 해외 사례
- [확인] **Citi Bike(뉴욕)**: 도크에 "렌치 아이콘" 물리 버튼이 있어, 반납 시 누르면 도크 표시등이 빨간불로 바뀌어 다른 이용자가 그 자전거를 못 뽑게 잠근다 (정비사 순회 전까지 도크 단위로 격리). 별도 웹 vandalism 신고 폼도 운영. QR 손상 시엔 수동 번호입력으로 우회 가능.
- [확인] **Lime**: 지도 화면의 "!" 아이콘 → "damaged vehicle" 선택 → 신고 즉시 해당 차량이 **다른 이용자 지도에서 실시간으로 사라짐**(자기 신고가 즉각 반영되는 구조). 이는 "이용자 체크인이 시스템 상태를 즉시 바꾸는" 가장 적극적인 모델.
- [확인] **GBFS(General Bikeshare Feed Specification, 국제 표준)**: `station_status.json`에 `num_vehicles_disabled`(구 `num_bikes_disabled`), `num_docks_disabled` 필드가 표준으로 정의되어 있다. 즉 국제 표준 자체는 "이용가능"과 "고장/비활성"을 분리 노출하도록 설계돼 있으나, **사업자가 원치 않으면 이 필드를 생략(비공개) 가능**하도록 스펙이 허용한다. → 타슈가 이 필드를 공개하지 않는 것은 GBFS를 따르지 않거나(추정), 따르더라도 옵션 필드를 의도적으로 비공개했을 가능성(추정)을 보여준다.
- [추정, 업계 일반론] IoT 텔레매틱스 업계에서는 배터리 전압, 잠금 상태, 진동/가속도 센서, GPS 드리프트 등을 결합한 자가진단·예측정비가 일반적 관행이지만, **타슈 자전거에 구체적으로 어떤 센서가 탑재돼 있고 그 데이터가 어디까지 내부 관제시스템에서만 쓰이는지는 미확인**이다. GPS+스마트락 탑재는 확인됐으나 배터리/가속도 자가진단 여부는 확인 불가.

## 2. 정류장 단위 데이터(합계만)에서 파생 가능한 고장 신호 (논리적 도출)

전제: 우리가 얻는 것은 시계열 `parking_count(station_id, t)` 뿐이다. 개별 자전거 ID·잠금상태·대여시도(성공/실패) 로그는 원칙적으로 API에 없다. 아래 신호들은 이 제약 안에서 **논리적으로 도출 가능한 간접 신호**이며, 확정적 "고장 탐지"가 아니라 "의심 정류장 스코어링"에 가깝다.

| # | 신호 | 정의 | 탐지 가능성 | 한계·오탐 요인 |
|---|---|---|---|---|
| 1 | **정체 재고(Stagnant Inventory)** | 수요가 존재하는 것으로 알려진 정류장(과거 회전 이력 있음)에서 재고 절대량이 장시간(예: N일) 변동폭이 비정상적으로 작음 | 중 — 정류장 총량 시계열만으로 근사 가능 | 개별 대수 특정 불가. 총량이 안 줄어도 실제로는 반납=대여로 순환 중일 수 있어 "고장 재고"와 "저수요"를 구분 못함. 계절/요일 편향 보정 필요 |
| 2 | **만차인데 회전율 0(Full-but-Frozen)** | 거치대가 포화 상태(`parking_count`≈용량)인데 시간대별 순감소(대여로 인한 감소)가 과거 동시간대 baseline 대비 현저히 낮음 | 중~강 — 과거 대비 이상탐지(anomaly detection)로 상대적으로 견고 | 심야·외곽 등 원래 저수요 시간대와 혼동 가능. baseline이 없는 신규 정류장엔 적용 불가 |
| 3 | **비대칭 증감 패턴** | 반납이 몰리는 시간대(예: 퇴근 후)에 총량이 기대만큼 늘지 않거나, 반대로 대여 몰리는 시간대에 기대만큼 안 줄어드는 정류장이 반복적으로 나타남 | 약~중 — 순수 통계적 이상치로만 잡히며 원인(고장/수요변화/공사 등)은 구분 불가 | False positive 매우 높음. 단독 신호로는 실전 배포 부적합, 반드시 보조 신호와 결합 필요 |
| 4 | **피어 비교(동종 정류장 대비 아웃라이어)** | 유사한 수요 패턴(위치·시간대 프로파일 유사)을 가진 인접/유사 정류장 군집과 비교해 유독 회전율이 낮은 정류장을 상대적으로 표시 | 중 — 정류장 수가 충분히 많을 때만 통계적으로 유의미 | 소도시(타슈처럼 정류장 수가 적은 시스템)는 비교군 자체가 부족해 신뢰도 낮음 |
| 5 | **반대 현상: 미반납 오표기(재고 과소 표기)** | 정상 반납했음에도 GPS/락 인식 실패로 정류장 재고가 실제보다 적게 표시 (기지 민원 사례) | 약 — station 데이터만으로는 아예 감지 불가, 사용자 신고(체크인)에 전적으로 의존 | 이 신호는 station API로는 원천적으로 관측 불가능하다는 점을 명시할 필요 있음 |
| 6 | **즉시 재대여 실패 반복** | 반납 직후 짧은 시간 내 같은 정류장에서 대여 시도가 급감/실패하는 패턴 | 매우 약함 — station 합계 데이터엔 "대여 시도(실패 포함)" 자체가 없음 | **station-only 데이터로는 원리적으로 탐지 불가.** 반드시 이용자 체크인 또는 대여이력 API(있다면)와 결합해야 성립하는 신호 |

**종합 평가**: station 합계 데이터만으로 도출 가능한 신호(#1~4)는 전부 **"의심(suspect) 스코어"** 수준이며, 확정 판정에는 부적합하다. 신호 강도 순위는 대체로 #2(만차-무회전) > #1(정체재고) > #4(피어비교) > #3(비대칭증감)이며, #5·#6은 station 데이터만으로는 원천적으로 도달 불가능한 영역으로 — **이것이 바로 "체크인" 개념(3장)이 구조적으로 필요한 이유**다. 즉, 순수 시스템 자동탐지는 "어디를 의심할지"까지만 좁혀줄 뿐, "실제로 고장인지"는 사람(이용자/정비사)의 확인 없이는 확정할 수 없다.

## 3. "체크인" 개념 설계 옵션 3가지

공통 상태집합을 먼저 정의: `normal(정상)` → `suspected(의심)` → `confirmed(고장확정)` → `resolved(수리완료)` → `normal` (재순환), 및 `dismissed(오신고 기각)` / `auto-cleared(자동해제)` 분기.

### (a) 이용자 체크인/신고형
**워크플로**
1. 사용자가 정류장/자전거에서 문제 발견 → (우리) 앱에서 QR 스캔 또는 정류장명+자전거번호 수동입력
2. 문제유형 선택(잠김불가/배터리없음/외관파손/GPS 미인식 등) + 사진 첨부(선택, 신뢰도 가중치용)
3. 제출 → 상태 `pending(개별 신고)` 로 기록
4. 동일 대상에 대해 **독립 신고 N건 누적**(짧은 시간창 내 동일 IP/기기 중복 배제) → `suspected` 로 격상, 지도에 "확인 필요" 뱃지 노출
5. 이후 신고가 더 들어오거나(강화) / 다른 이용자가 "정상 사용했음" 반박 체크인(약화) → 신뢰도 점수 재계산
6. 정비사 체크인(b) 또는 시간경과 자동판정으로 `confirmed`/`dismissed` 확정

**상태 전이**: `normal → (신고 1건) pending → (임계치 N건 도달) suspected → (교차검증) confirmed | dismissed → (수리보고 or 정비사 체크인) resolved → normal`

**장점**: 가장 빠르게 시작 가능(운영기관 협조 불필요), 실시간성 최고. **한계**: 콜드스타트(초기 이용자 수 부족), 악의적 허위신고(경쟁자·장난) 리스크, 신뢰도 가중치 알고리즘 설계 필수.

### (b) 정비사 체크인형
**워크플로**
1. 정비사가 순회 일정에 따라 정류장 도착 → 정비앱에서 **GPS 기반 도착 체크인**
2. 각 자전거 QR/번호 스캔 → 상태 선택(정상 / 경미결함-현장수리 / 중대결함-즉시회수)
3. 조치 내용·부품 사용 기록 → 완료 시 타임스탬프와 함께 `serviced` 도장
4. 시스템 반영: 해당 정류장·자전거의 "최종 정비 확인일" 갱신 → **이용자 체크인(a)의 신뢰도 보정 기준선**으로 활용(예: 정비사가 방금 정상 확인한 자전거에 대한 사용자 오신고는 가중치 하향)

**상태 전이**: `flagged(정비대상 지정) → assigned(배정) → arrived(현장도착, GPS체크인) → inspecting(점검중) → returned_to_service(정상복귀) | withdrawn(회수) | parts_pending(부품대기)`

**장점**: 신뢰도가 가장 높은 데이터 소스(정답에 가장 가까움), 다른 두 체크인 방식의 검증 기준(ground truth) 역할 가능. **한계**: 우리가 정비 인력을 통제할 수 없다면(= 운영기관 소속 정비사) 이 체크인은 **운영기관 제휴 없이는 구현 불가능** — 4장 배포형태 논의와 직결.

### (c) 시스템 자동체크인형 (AI/이상탐지 기반)
**워크플로**
1. 2장의 신호(#1~4 등)를 결합한 이상탐지 파이프라인이 정류장별 스코어 산출
2. 스코어 임계치 초과 → `ai_flagged` 마킹. **주의: 이 단계에서는 절대 "고장 확정"으로 사용자에게 노출하지 않고 "확인 필요" 수준의 낮은 신뢰도 뱃지로만 표시** (오탐 리스크가 구조적으로 크기 때문, 2장 참조)
3. 교차검증 트리거: (a) 이용자 체크인 누적 또는 (b) 정비사 체크인과 일치 시 → `confirmed`로 승격
4. 반증 트리거: 이후 해당 정류장에서 정상적 대여 재개(총량 정상 순환 관측) 시 → `auto-cleared`로 자동 해제, 별도 사람 개입 없이 `normal` 복귀

**상태 전이**: `normal → (이상탐지 스코어>임계치) ai_flagged → (교차검증 성공) suspected/confirmed → (수리 or 자연회복) resolved/auto-cleared → normal`

**장점**: 사람 개입 없이 24시간 전 정류장 스캔 가능, (a)(b)가 데이터 희소한 신규 지역/시간대의 공백을 메움. **한계**: 2장에서 평가했듯 station-합계 데이터만으로는 신호 강도가 약함(특히 #5, #6 유형은 원천 탐지 불가) → **단독으로는 신뢰 못 하고 반드시 (a)/(b)와 결합하는 하이브리드 구조로만 실용성 확보**.

### 통합 권장 모델
세 체크인을 하나의 상태머신으로 묶어 "다중 소스 신뢰도 점수(trust score)" 방식으로 운영하는 것이 현실적이다: 자동탐지가 후보를 좁히고(리콜 확보) → 이용자 신고가 즉시성을 제공하고(커버리지 확보) → 정비사 확인이 최종 판정(정확도 확보)하는 3단 파이프라인. 이는 안전신문고류의 순수 시민신고 단일소스 모델(아래 4장 참조)이 갖는 낮은 신뢰도 문제를 구조적으로 보완한다.

## 4. 법·운영 제약 및 현실적 배포 형태 비교

### 제약 조건
- [확인] 공공데이터포털 기준, 대부분의 공공데이터는 무료 및 상업적 활용까지 허용되나 "일부 데이터는 별도 조건"이 있을 수 있음 — 타슈 오픈API는 앱 내 개별 신청 방식이라 **표준 공공데이터포털 라이선스와 별개의 자체 이용약관 존재 가능성 있음 (추정, 재확인 필요)**.
- [확인] `api-token` 헤더가 필수인 구조 자체가, 운영기관이 접근을 통제·모니터링하고 있음을 의미한다 → 정식 토큰 신청 없는 우회 수집(스크레이핑 등)은 이용약관 위반 소지가 크며, 서비스 차단·법적 리스크로 이어질 수 있다 [추정: 제재 수위는 미확인].
- [추정] 우리가 산출한 "고장 의심" 마킹을 마치 공식 데이터인 것처럼 재배포하면 이용자에게 오인을 유발하거나 운영기관 신용을 훼손했다는 주장이 제기될 수 있음 → **반드시 "비공식/추정치" 고지가 UI상 명확해야 함** (법률 자문 별도 필요).
- [확인] 선례: 대구광역시-카카오T바이크는 "가격 동결" 조건으로 **공공 거치대 무상 사용 협약**을 체결한 바 있음 — 지자체가 민간과 데이터/인프라를 공식 제휴하는 사례가 실제 존재한다.
- [확인] 행정안전부가 2024년 "전국 통합데이터 개방 확대 사업"에 공영자전거 대여현황 등 4종 데이터를 포함해 표준화 개방을 추진 중이라고 발표(세부 필드는 미공개) → 향후 국가 차원에서 실시간 데이터 표준화 움직임이 있다는 방향성 신호이나, 고장 대수 필드 포함 여부는 확인 불가.

### 배포 형태 비교

| 형태 | 설명 | 장점 | 단점 / 리스크 | 법적 안전성 |
|---|---|---|---|---|
| **① 별도 보정 앱/웹 (비공식 서드파티)** | 공식 오픈API 재가공 + 크라우드소싱 신고 오버레이를 독립 서비스로 배포 | 즉시 시작 가능, 운영기관 승인 불필요, 빠른 반복(iteration) | 자전거 단위 식별 불가(정류장 합계만 있어 정밀도 한계), 콜드스타트 문제, "타슈 공식 아님" 고지 필수, 대량 API 호출 시 차단 리스크 | 중간 — 명확한 비공식 고지·공공데이터 이용약관 준수 시 상대적으로 안전 |
| **② 운영기관 제휴 제안** | 데이터 공유 MOU 체결, 우리 크라우드소싱 신고를 공식 유지보수 티켓 시스템(정비사 체크인 b)에 직결 | 공식 앱 표기 개선에 직접 기여, 정비사 체크인(가장 신뢰도 높은 데이터) 접근 가능, 법적 리스크 최소 | 지자체·공기업 협상·조달 절차로 소요기간 김, 초기 실증(트랙레코드) 없이는 제휴 성사 어려움 | 가장 안전 — 단, 성사까지 시간 소요 |
| **③ 공익 대시보드 (투명성/연구 목적)** | 정류장별 이상탐지 결과·통계를 시각화해 시민·언론·감사 자료로 공개 (개별 행동유도 없음) | 법적으로 가장 안전(개인 대상 서비스가 아닌 통계 공개), 데이터저널리즘/시민단체 협업 여지, 실시간성 요구 낮음 | 실사용자의 즉각적 불편(대여 실패) 해소엔 간접적 효과만 있음, 참여 유인이 약해 크라우드소싱 데이터 확보가 느릴 수 있음 | 매우 안전 |

**현실적 로드맵 제안(추정, 전략적 판단)**: ③(공익 대시보드)이나 ①(보정 앱 MVP)로 트랙레코드와 데이터를 먼저 축적한 뒤, 그 실증 결과를 근거로 ②(운영기관 제휴)를 제안하는 단계적 접근이 협상력 측면에서 유리하다. 정비사 체크인(3장 b)은 사실상 ②가 성사돼야만 구현 가능한 옵션이므로, ①③ 단계에서는 이용자 체크인(a)과 시스템 자동체크인(c)의 하이브리드로 먼저 최소기능제품을 검증하는 것이 순서상 타당하다.
