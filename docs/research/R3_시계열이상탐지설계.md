**설계 임팩트**: 정류장 ~1,000개 × 1~5분 폴링이면 하루 약 28.8만~144만 행이 쌓인다. 정류장이 1,500개로 늘면 최대 하루 216만 행 수준 — 단일 서버 PostgreSQL/TimescaleDB로 충분히 감당되는 규모이며, 굳이 InfluxDB/Kafka 클러스터급 인프라가 필요한 볼륨은 아니다. 이 규모 산정이 4번 파이프라인 선택의 근거가 된다.

## 1. 후보 모델 비교

| 모델군 | 콜드스타트 | 계절성(출퇴근/주말/날씨) | 실시간 추론 비용 | 해석가능성 | 비고 |
|---|---|---|---|---|---|
| **STL 분해 + 잔차** | 최소 2~3주기(보통 2주 이상) 필요하나, 정류장 클러스터 공통 계절 성분을 부트스트랩하면 즉시 적용 가능 | 일간/주간 계절성은 강함, 날씨 등 외생변수는 별도 처리 필요 | 매우 낮음(분해는 배치, 잔차 비교는 O(1)) | 매우 높음(추세/계절/잔차 시각적 분리) | 정류장별 개별 모델이 1,000개 필요 → 계산 자체는 가볍지만 운영(모델 수) 부담. 스파이크에 민감(로버스트 STL 필요) |
| **ARIMA/SARIMA** | STL과 유사하게 데이터 축적 필요, 정상성 가정 위반 시 성능 저하 | SARIMA는 계절 항을 명시적으로 모델링 가능(주기=하루 288스텝@5분) | 낮음~중간(재적합 비용이 큼, 온라인 업데이트 비native) | 높음(계수 해석 가능) | 정류장 수만큼 파라미터 튜닝이 필요해 1,000개 정류장 개별 SARIMA는 운영비용 큼 → 클러스터 단위 권장 |
| **Prophet** | 휴일/이벤트 사전 지정 가능해 콜드스타트에 비교적 강함(며칠 데이터로도 대략적 계절성 추정) | 날씨를 외생 regressor로 손쉽게 추가 가능, 다중 계절성(일/주) 기본 지원 | 낮음(학습은 배치, 추론은 가벼움) | 높음(성분 분해 그래프 제공) | 정류장별 개별 fit 필요 시 1,000개 모델 재학습이 오래 걸림(수 분~수십 분/일) — 이상탐지 목적엔 다소 무거운 편 |
| **Isolation Forest (unsupervised ML)** | 매우 빠름(피처만 있으면 라벨 없이 즉시 학습 가능) — 신규 정류장도 전역 모델에 즉시 태울 수 있음 | 시간대/요일/날씨를 피처로 넣으면 간접 반영, 명시적 계절 모델링은 아님 | 매우 낮음(트리 앙상블 추론) | 중간(feature importance/SHAP으로 설명 가능하나 STL만큼 직관적이지 않음) | 전역(global) 모델 하나로 전 정류장 커버 가능 → 신규 정류장 콜드스타트에 가장 유리 |
| **LightGBM 잔차 모델(지도 회귀)** | 라벨(정상 vs 이상)이 아니라 "다음 재고량 예측"을 회귀 타깃으로 삼으면 준지도 방식으로 즉시 학습 가능 | 시간/요일/날씨/정류장ID를 피처로 넣어 계절성+정류장 특성 동시 학습(글로벌 모델, station을 카테고리 피처로) | 낮음(추론 매우 빠름), 재학습은 일 단위 배치로 충분 | 중간(SHAP value로 설명 가능) | **1,000개 정류장을 pooled 데이터로 학습하는 글로벌 모델**이라 콜드스타트·계절성·비용 균형이 가장 좋음. 신규 정류장도 인접 정류장 피처로 즉시 예측 가능 |
| **LSTM-AE (오토인코더)** | 약함(정류장별 수천 스텝 이상의 정상 패턴 학습 필요, GPU 학습시간 소요) | 시퀀스 자체로 학습하므로 암묵적으로 반영되나 명시적 제어는 어려움 | 중간~높음(추론은 가볍지만 학습/재학습 비용 큼, 배치 재학습 필요) | 낮음(재구성 오차만 제공, 원인 해석 어려움) | 정상 패턴이 다양한 1,000개 정류장에 단일 AE를 쓰면 평균화되어 미세 이상 놓칠 위험, 정류장별 AE는 비현실적 |
| **TCN (Temporal Conv Net)** | LSTM보다 학습 빠르고 병렬화 잘 됨이나 여전히 수백~수천 샘플 필요 | 긴 receptive field로 주간 패턴까지 포착 가능 | 중간(추론 빠름, GPU 없어도 CPU에서 무난) | 낮음~중간 | 예측 기반(잔차) 방식으로 쓰면 LightGBM 대비 뚜렷한 이득이 크지 않음(정형 피처 문제라 트리 계열이 보통 더 강함) |
| **Transformer 계열(TFT 등)** | 매우 무거움(다중 정류장 공동학습 시 데이터는 충분해지지만 학습 인프라·튜닝 비용 큼) | 다중 계절성/외생변수(날씨) 결합에 가장 강력 | 높음(GPU 권장, 실시간 저비용 요건과 상충) | 낮음(attention map으로 부분적 설명 가능하나 운영자 친화적이지 않음) | 이 문제 규모(정형 저차원 피처, ~1,000개 시계열)에서는 과설계 가능성 높음. 데이터가 수개월~1년 이상 쌓이고 다변량 상호작용이 중요해질 때 2단계 고도화로 고려 |

**결론(1절)**: 콜드스타트·계절성·해석가능성·비용을 모두 만족하는 단일 모델은 없다. **"통계 baseline(STL 잔차) + 지도 ML(LightGBM 잔차, 글로벌 모델) + 규칙 기반 정체탐지"의 하이브리드**가 이 문제엔 가장 실용적이며, DL은 데이터가 충분히 쌓인 뒤(6개월 이상) 정확도 개선 목적의 2단계 옵션으로 남긴다.

## 2. 피처 엔지니어링

전제: 개별 자전거 ID 없이 **정류장별 (timestamp, 보유대수, 거치대수)** 스냅샷만 있다고 가정. [추정 — 실제 스키마 미확인이므로 필드명은 가정]

### 2.1 기본/파생 피처
- **점유율**: `occupancy_rate = bikes / capacity` (capacity=0 정류장은 별도 예외 처리)
- **재고 변화량**: `delta_t = bikes_t - bikes_{t-1}` (폴링 간격 고정 가정, 결측 시 시간 가중 보정)
- **재고 변화율**: `pct_change_t = delta_t / capacity`
- **롤링 통계**: 15/30/60분 윈도우의 이동평균, 이동표준편차, 이동 기울기(선형회귀 slope)
- **회전율(turnover) 추정** — *중요 caveat*: 개별 ID가 없으므로 진짜 대여/반납 횟수는 알 수 없다. 폴링 간격보다 짧게 왕복(대여 후 곧 반납, 혹은 그 반대)이 일어나면 스냅샷에 안 잡혀 **과소추정**된다.
  - proxy 1: `turnover_proxy = Σ|delta_t|` (윈도우 내 절대변화량 합) / 2 — "최소 이만큼의 이동은 있었다"는 하한선(lower bound)
  - proxy 2: `sign_change_count` (증가↔감소 전환 횟수) — 활발한 대여소 여부 판별
  - 데이터가 허용되면 "대여이력" 배치 데이터셋과 조인해 실제 트립 카운트로 보정 [추정 — 결합 가능 여부 미검증]
- **정체 시간(stagnation duration)**: `time_since_last_nonzero_delta` — 마지막으로 대수가 변한 시점부터 경과 시간(3번 항목의 핵심 피처)
- **인접 정류장 대비 상대 수요**: 위경도로 반경 500m~1km 내 정류장 클러스터를 구성(k-NN 또는 grid), 해당 클러스터의 평균 `pct_change`/`occupancy_rate` 대비 z-score. "도시 전체가 조용한 새벽" vs "이 정류장만 멈춘 것"을 구분하는 핵심 피처
- **시간/요일/캘린더**: `hour`(sin/cos 순환 인코딩), `dow`, `is_weekend`, `is_holiday`(공휴일 API 또는 정적 테이블), `is_rush_hour`(07-09, 18-20 등 대전 통근 패턴 기준)
- **날씨 결합**: 기상청 API(강수량/기온/풍속)를 시간 단위로 조인 — 비/폭염/한파일 때는 낮은 회전율이 "정상"이므로 잔차 모델의 외생변수로 필수. [추정 — 별도 API 연동 필요, 본 조사에서 기상청 API 자체는 미검증]
- **결측/장애 플래그**: 폴링 실패(네트워크 오류, HTTP 5xx) 시 forward-fill 대신 `is_missing` 플래그를 남기고 gap 길이를 별도 피처로 — API 장애를 정류장 고장으로 오판하지 않도록 함
- **정류장 메타 피처**: `capacity`(거치대 수), 신설 여부(`days_since_launch`, 콜드스타트 처리용), 위치 유형(역세권/주거/공원 등은 외부 라벨링 필요 — [추정])

## 3. "고장 의심" 판정 로직 — 하이브리드 룰 (제안)

두 축을 독립적으로 스코어링한 뒤 결합한다.

### 축 A — 예측 잔차 기반 (재고 이상)
1. 베이스라인 예측값 `ŷ_t` 생성: LightGBM(또는 STL+SARIMA 백업)으로 `occupancy_rate_t`를 시간/요일/날씨/인접정류장 피처로 예측
2. 잔차 `r_t = y_t - ŷ_t`
3. **로버스트 z-score**(평균/표준편차 대신 median/MAD 사용 — 이상치에 강건):
   `z_t = 0.6745 * (r_t - median(r)) / MAD(r)` (윈도우: 최근 14일 동일 시간대)
4. **지속성 필터(CUSUM/EWMA)**: 단발성 튐(재배치 트럭 방문 등)을 걸러내기 위해 `|z_t|>3` 이 **연속 N회(예: 15분 이상, 폴링 간격에 따라 3~15회)** 지속될 때만 이상 후보로 승격 — CUSUM `S_t = max(0, S_{t-1} + z_t - k)` 방식으로 누적 편차 추적

### 축 B — 회전율 정체 기반 (기능 이상/고장 의심)
1. `stagnation_minutes = time_since_last_nonzero_delta`
2. 정류장별 **적응형 임계값**: 전역 고정값(예: "2시간") 대신, 해당 정류장·해당 시간대(hour bucket)의 과거 "정체 지속시간 분포"에서 상위 percentile(예: 95th~99th)을 임계값으로 사용 — 원래 한산한 정류장과 번화가 정류장의 정상 정체시간이 다르기 때문
3. **경계 조건 분리**:
   - `bikes=0` (완전 소진) & 정체 지속 → "재고 소진"이지 고장이 아닐 수 있음 → 별도 카테고리(재배치 필요 알람)
   - `bikes=capacity` (완전 만차) & 정체 지속 → 반납 불가 의심(거치대 고장/포화) → 고장 의심 가중치 ↑
   - `0<bikes<capacity` & 장시간 정체 → 거치대 잠금장치/센서 고장, 혹은 통신 두절 의심 → 고장 의심 가중치 최고
4. 인접 정류장은 정상 회전 중인데 해당 정류장만 정체 → 국지적(local) 이상으로 확정, 가중치 추가 상향(축 A의 상대수요 피처와 연동)

### 결합(하이브리드 스코어)
```
confidence = w1 * sigmoid(z_score_normalized)          # 축 A: 잔차 이상
           + w2 * sigmoid(stagnation_ratio)             # 축 B: 정체 비율(실제/임계값)
           + w3 * neighbor_divergence_score             # 인접 대비 국지성
           - w4 * data_quality_penalty                  # 결측/API 장애 페널티
```
- 초기 가중치는 도메인 규칙으로 설정(예: w1=0.35, w2=0.4, w3=0.2, w4=0.05), 이후 3-3에서 다룰 약라벨(신고 이력)로 로지스틱 회귀/Platt scaling을 붙여 `confidence`를 실제 확률처럼 보정(calibration)
- **등급화**: `confidence ≥ 0.8` → 즉시 알람(현장 출동), `0.5~0.8` → 대시보드 관찰 목록, `<0.5` → 로그만 적재
- **임계값 산출 방법론**: 처음엔 percentile 기반(예: 전체 스코어 분포 상위 0.5%를 알람) → 신고 데이터가 쌓이면 precision@k(3-5절)로 임계값을 재보정하는 **2단계 캘리브레이션** 방식을 제안

## 4. 파이프라인

### 4.1 수집(폴링)
- 스케줄러: Airflow(운영 성숙 시) 또는 초기엔 단순 cron + 경량 워커(예: Python `apscheduler`)로 시작 — 정류장 1,000개 규모면 과설계 지양
- 폴링 주기: 1~5분(요구사항대로), station 목록은 캐싱하고 개별 station 상세는 벌크 엔드포인트가 있다면 1콜로 전체 수집(효율), 없다면 정류장별 N콜 병렬 요청 + rate limit 준수
- `api-token` 재발급/만료 대응 로직, HTTP 4xx/5xx 재시도(backoff), 실패 시 `is_missing` 플래그로 gap 명시
- Idempotent upsert(동일 timestamp 중복 폴링 방지)

### 4.2 저장 — TSDB 선택 비교

| 항목 | TimescaleDB | InfluxDB | Plain PostgreSQL |
|---|---|---|---|
| 아키텍처 | PostgreSQL 확장(관계형+시계열 하이브리드) | 전용 시계열 NoSQL, 자체 쿼리언어(Flux/InfluxQL) | 순수 관계형 |
| 쓰기 성능(대량 IoT) | 양호, 고카디널리티(디바이스 수 많음)에서 압축·인덱싱으로 상대적 강점(자료상 ~3.5배 우위 언급) [확인: 웹 출처 종합] | 초고빈도 순수 시계열 쓰기에 최적화, out-of-box 처리량 우위 [확인: 웹 출처 종합] | 별도 시계열 최적화 없음, 대량 시계열엔 인덱스 비대화 위험 |
| 조인/메타데이터 | SQL로 정류장 메타(좌표·용량), 신고 라벨, 날씨 테이블과 자유롭게 JOIN 가능 | 조인 약함, 메타데이터는 태그로 제한적 관리 | 조인은 자유롭지만 시계열 전용 기능(continuous aggregate, 압축) 없음 |
| 생태계/러닝커브 | SQL 그대로 사용, pandas/SQLAlchemy 등 파이썬 생태계와 직결 | Flux/InfluxQL 별도 학습 필요, Python 클라이언트도 있으나 SQL 이질적 | 가장 친숙하나 확장성 기능 부재 |
| 이 프로젝트 규모(하루 최대 ~200만 행, 정류장 ~1,500개) | 무리 없이 처리, hypertable+continuous aggregate로 집계 쿼리 최적화 | 과할 정도로 충분한 성능(오히려 오버스펙 가능성) | 압축·자동 파티셔닝 없이도 초기엔 돌아가지만 장기 데이터 누적 시 성능 저하 우려 |

**선택**: **TimescaleDB** — 정류장 메타(좌표/용량), 신고 라벨(약라벨), 날씨 조인, 모델 결과 저장을 전부 SQL 한 곳에서 처리할 수 있다는 점이 이 프로젝트(고장 판정에 다중 소스 조인이 필수)에서 결정적이다. 순수 쓰기 처리량 우위인 InfluxDB의 이점은 이 데이터 볼륨(초당 수십~수백 행 수준)에서는 체감되지 않는다.

### 4.3 학습/추론 주기
- **실시간 추론**: 매 폴링마다 축 A(사전 학습된 LightGBM/STL 잔차) + 축 B(정체시간 룰)을 즉시 계산해 `confidence` 산출 — 무거운 재학습 없이 저장된 모델로 O(ms) 추론
- **배치 재학습**: 일 단위(새벽 유휴시간)로 LightGBM 재적합, 주 단위로 STL 계절성분 재추정, 정류장별 정체시간 percentile 임계값도 주 단위 갱신
- **드리프트 감지 및 재학습 트리거**:
  - 입력 분포 드리프트: PSI(Population Stability Index) 또는 KS-test로 최근 윈도우 vs 학습 윈도우 비교
  - 잔차 분포 드리프트: 잔차의 평균/분산이 유의하게 이동하면(예: 계절 전환, 신규 정류장 대량 추가 시점인 1,000→1,500 확장 시기) 강제 재학습 트리거
  - `river`의 `ADWIN`(Adaptive Windowing) 같은 온라인 드리프트 탐지기를 축 A 옆에 보조로 붙여, 배치 재학습 사이 갭에서도 급격한 변화를 조기 포착

## 5. 평가 방법 (라벨 희소 환경)

### 5.1 약라벨링(weak labeling)
- **신고 데이터 매칭**: 고객센터 민원/앱 신고, 재배치·정비팀 출동 이력을 (정류장, 시각) 이벤트로 확보 → 해당 시각 **± N시간 윈도우**를 positive 근사 라벨로 확장(정확한 고장 발생 시점은 신고 시점보다 앞설 가능성이 높으므로 윈도우를 비대칭으로, 신고 이전 쪽을 더 넓게)
- **PU-learning 관점**: "라벨 없는 = 정상"이 아니라 "라벨 없는 = 확인 안 됨(unlabeled)"으로 취급 — 신고 안 된 실제 고장이 존재할 수 있다는 전제하에 순수 negative로 취급하지 않는 PU-learning 기법(예: 신뢰도 낮은 negative는 학습에서 다운웨이트) 적용

### 5.2 지표
- **Precision@k**: 하루 상위 k건(예: confidence 상위 20건)의 알람 중 실제 신고/출동과 매칭되는 비율 — 운영팀이 감당 가능한 알람 수(k)로 실제 유효성을 측정하는 데 가장 실무적인 지표
- **Recall@매칭 신고건수**: 확보된 신고 라벨 중 모델이 사전(또는 근접 시점)에 알람을 울린 비율
- **Lead time**: 모델 알람 시각과 실제 신고 접수 시각의 차이(음수면 선제 탐지 성공) — 이상탐지 시스템의 실질 가치를 보여주는 핵심 지표
- **알람량/일(alert budget)**: precision만 보면 임계값을 극단적으로 올려 알람을 0건으로 만들 수 있으므로, "하루 X건 이내" 제약 하의 recall/precision을 같이 봐야 함
- **False positive 감사 로그**: 운영자가 알람을 "정탐/오탐" 클릭으로 피드백하게 하여 점진적으로 진짜 라벨을 축적(human-in-the-loop) — 초기 몇 달은 이 피드백 루프 자체가 라벨 획득의 주 채널

### 5.3 배포 전략
- **섀도 모드**: 신규 모델/임계값을 실제 알람 발송 없이 로그만 쌓아 기존 룰(또는 이전 버전)과 나란히 비교한 뒤 전환
- **온라인 A/B**: 정류장 그룹을 나눠 임계값 변형을 비교(단, 지리적 인접 정류장 그룹 간 교차오염 주의)

## 최종 추천 스택 (명시적 픽)

1. **저장소**: **TimescaleDB**(PostgreSQL 확장) — 메타데이터·신고 라벨·날씨와의 SQL 조인 필요성이 이 프로젝트의 핵심 요구이므로 InfluxDB보다 우위로 판단
2. **베이스라인 예측 모델**: **STL(statsmodels) 분해 + 로버스트 잔차 z-score** — 정류장/클러스터 단위로 빠르게 배치 적용, 해석가능성 최우선 확보
3. **메인 예측 모델(축 A)**: **LightGBM 잔차 회귀**, 전 정류장 pooled 학습(station ID·시간·요일·날씨·인접정류장 피처 포함) — 콜드스타트·계절성·비용·해석가능성의 최적 균형점으로 **주력 모델로 픽**
4. **정체탐지(축 B)**: 정류장·시간대별 적응형 percentile 임계값 기반 규칙 로직 (3절 상세)
5. **실시간 보조 레이어**: **river**(`HalfSpaceTrees` + `ADWIN`)로 배치 재학습 사이의 급변 상황을 온라인으로 보완
6. **통합 프레임워크**: **darts**를 실험/파이프라인 표준 레이어로 채택 — STL/SARIMA/LightGBM/향후 TCN까지 동일 인터페이스로 교체·비교 가능, `darts.ad`로 스코어러-탐지기-집계기 파이프라인 구성
7. **DL(LSTM-AE/TCN/Transformer)은 2단계 고도화 옵션으로 유보** — 데이터가 6개월~1년 이상 누적되고, 규칙+LightGBM 하이브리드로 못 잡는 복합 패턴(다변량 상호작용)이 실증적으로 확인될 때 도입 검토
