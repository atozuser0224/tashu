# 타슈(Tashu) API 분석

## 0. 요약 (TL;DR)

- **[확인]** 타슈는 최소 **두 개의 서로 다른 오픈API 등록 경로**를 갖고 있다: (A) 타슈 자체가 운영하는 직접 API `bikeapp.tashu.or.kr:50041`, (B) 공공데이터포털(data.go.kr)이 자체 게이트웨이로 중계하는 `대전광역시_타슈정보`.
- **[확인]** 실시간 정류소 API는 **정류소 단위 집계 데이터만** 제공한다. 자전거 개별 ID, 고장/파손 여부, 거치대 총 용량(도크 수) 필드는 **모두 존재하지 않는다.**
- **[확인]** 개별 "자전거번호"(bike ID) 필드는 **연 1회 갱신되는 대여이력 CSV**에는 존재하지만, 실시간 API에는 노출되지 않는다.

### 인증(토큰 발급) 절차 [확인 - 공지사항 원문 기준]
1. 타슈 앱 로그인 → 사이드메뉴 `자전거정보 > OpenAPI` 메뉴에서 신청
2. 실무담당자가 제공 여부를 **수동으로** 검토
3. 승인/반려 결과를 신청인에게 통보
4. 승인 시 `인증키` 발급 → 이후 모든 요청의 `api-token` 헤더에 삽입
5. 호출 방식: GET, 응답: JSON

**소요기간 및 정확한 일일 호출 한도는 공개되어 있지 않다.** 공지 원문은 "일일 호출 한도를 지정하며, 과도한 트래픽 시 차단될 수 있음"이라고만 명시([확인] — 문구는 확인, 구체적 숫자는 [미확인]). 실제로 이 API를 인프라에 편입시킨 개발자들 다수가 "수동승인, 소요기간 불명이므로 주 경로로 삼지 않는다"고 판단해 대체 데이터 소스를 보조로 병행하고 있었다(예: `atozuser0224/KnuSoftHackerton` 프로젝트 계획 문서).

참고로 **별도 경로**인 data.go.kr `대전광역시_타슈정보`(15109253)는 데이터포털 자체 페이지에 "심의유형: 자동승인, 개발계정 10,000회/일"이라고 명시돼 있다 **[확인 - 문구]**. 다만 이것이 `bikeapp.tashu.or.kr` 직접 API와 완전히 동일한 데이터·서버인지, 아니면 data.go.kr이 별도로 캐싱/중계하는 구조인지는 **[불명]** — 데이터셋 번호·심의유형(자동 vs 수동)이 서로 달라 별개 시스템으로 운용되는 것으로 보인다.

## 3. 실시간 정류소(station) 응답 스키마

`GET https://bikeapp.tashu.or.kr:50041/v1/openapi/station` (Header: `api-token`)

### 실제 라이브 curl 검증 (토큰 없음/무효 토큰 공통)
```
HTTP/1.1 400 Bad Request
Server: nginx
Allow: GET, HEAD, OPTIONS
Vary: Accept
X-Frame-Options: DENY

{"error_code":"1001","message":"권한이 없습니다.","message_en":"You do not have permission.","message_cn":"权限不存在。","status_code":400}
```
`Allow: GET, HEAD, OPTIONS` / `Vary: Accept` 헤더는 Django REST Framework(DRF)의 특징적 서명이다 **[확인 - 간접 근거]**. 이는 공지사항에 언급된 `count/next/previous/results` 페이지네이션 래퍼가 DRF 기본 페이지네이션 스타일과 일치함을 뒷받침하지만, 실제 200 응답에서 이 래퍼 필드를 직접 관찰하지는 못했다 **[추정]**.

### 필드 표 (results 배열 내 원소, 6개 이상 독립 리포지토리에서 동일 확인됨)

| 필드 | 타입(관찰값) | 설명 | 확인 상태 |
|---|---|---|---|
| `id` | string, 예 `"ST0003"` | 정류소 ID | **[확인]** |
| `name` | string | 정류소명(한글) | **[확인]** |
| `name_en` | string(추정) | 정류소명(영문) | **[추정]** — 공지 원문엔 언급되나 실제 GitHub 샘플/파서 코드에는 등장 안 함(다들 `name`만 사용) |
| `name_cn` | string(추정) | 정류소명(중문) | **[추정]** — 위와 동일 사유 |
| `x_pos` | string, 예 `"36.348446"` | **위도(latitude)** | **[확인]** — 실제 값이 36.x대(대전 위도)임을 다수 실측 데이터로 확인. `x_pos`인데 위도라는 이름-의미 불일치를 실제 개발자들이 소스코드 주석으로 명시 지적함("x_pos는 위도, y_pos는 경도로 혼용됨") |
| `y_pos` | string, 예 `"127.390052"` | **경도(longitude)** | **[확인]** |
| `address` | string | 도로명/지번 주소 | **[확인]** |
| `parking_count` | integer, 예 `0`, `7` | **현재 대여 가능한 자전거 수(집계값)** | **[확인]** |
| `count`/`next`/`previous` | — | 페이지네이션 래퍼 후보 | **[추정]** — 서버가 DRF임은 확인했으나 실제 응답 캡처로 직접 관찰 못함 |

### 실측 샘플 JSON (GitHub `Jeric1223/TASHU-OPTIMAL-ROUTE-FINDER`, 5분 주기 크론으로 캡처한 실데이터)

### 핵심 질문 — 개별 자전거 ID / 고장 여부 필드 존재하는가?

**[확인 — 존재하지 않는다]**
- station 실시간 API 응답에는 개별 자전거 고유번호, 개별 자전거 상태(정상/고장/파손), "현재 대여중" 여부 등 **자전거 단위 필드가 전혀 없다.** 오직 정류소 단위 집계 수량인 `parking_count` 하나만 제공된다.
- 이는 Java(`DayGong`, `None-Step-BE`), Node.js(`SeongBinBin`, `Jeric1223`, `MkBaek0229`), Python(`Tyranno-Rex`, `pcu-tashu-dev`), Kotlin/Dart(`jihyun1123`, `leejieun0000`) 등 언어·플랫폼이 전혀 다른 6개 이상의 독립 프로젝트에서 **일관되게** `id/name/x_pos/y_pos/address/parking_count` 6개 필드만 파싱하고 있는 것으로 교차검증됨.
- `pcu-tashu-dev/server`(정류소 대여량 예측 서비스)조차 station API로는 정류소별 자전거 "수량"만 수집하고, 개별 자전거 추적은 시도조차 하지 않는다 — 애초에 API가 그런 데이터를 주지 않기 때문으로 보임 **[추정 - 정황 근거]**.
- **[확인]** 다만 개별 "자전거번호" 필드 자체는 타슈 시스템 내부에 존재한다 — data.go.kr **대여이력 CSV(15137219)**의 제공 컬럼에 `자전거번호`가 명시되어 있다. 즉 자전거 고유 ID는 시스템에 존재하지만, **실시간 오픈API로는 노출하지 않고 연 1회 배치로 내려주는 이력 파일에만 담겨 있다.**
- **[확인 - 미존재]** "고장 여부"/파손 상태 필드는 실시간 API 응답에도, 대여이력 CSV 컬럼 목록에도 어디에도 확인되지 않았다. 두 데이터 소스 모두에서 부재가 확인된 것이므로 이 결론의 신뢰도는 높다.
- **[확인 - 미존재]** 거치대 총 용량(도크 개수) 필드도 없다. `parking_count`(현재 있는 자전거 수)만 알 수 있고, 정류소가 "꽉 찼는지" 판단할 기준(최대 수용량)은 API에 없다. 2020년 위치현황 파일(15062798)조차 컬럼 목록에 거치대 수가 없다.

## 6. 한계점 / 리스크 목록

1. **공식 API 문서 부재**: Swagger/PDF 등 공개 명세가 웹에 없고, 앱 내부 승인 후 안내되는 구조 → 외부에서 사전 검증이 어려움 [확인]
2. **수동 승인·소요기간 불명**: 자동승인 없이 "실무담당자 검토"를 거쳐야 하며 실제 서드파티 개발자들이 이를 병목으로 지목함 [확인]
3. **개별 자전거 추적 불가**: 고유 ID·GPS 위치·고장상태 등 자전거 단위 정보가 실시간 API에 전무 → "고장 자전거 지도" 같은 서비스는 이 API만으로는 제작 불가능 [확인]
