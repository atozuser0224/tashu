# LLM 해석 레이어 설계 — 타슈 이상탐지 실시간 해석 (Claude API 기준)

> 조사 시점: 2026-07-21. 모델명·가격은 `platform.claude.com/docs/en/about-claude/pricing.md`를 직접 WebFetch로 재확인했다(스킬 캐시는 2026-06-24 기준이었으나 오늘 재확인한 값과 일치). **[확인]** / **[추정]** 태그로 구분한다.

## 0. 전제 및 아키텍처 위치

- **[확인]** 타슈 오픈API(`https://bikeapp.tashu.or.kr:50041/v1/openapi/station`)는 살아있고 `api-token` 헤더 없이는 401 상당의 `{"error_code":"1001"}`을 반환한다 — 이 조사에서 재검증하지 않고 기존 확인 사실로 채택.
- 이 문서가 다루는 LLM 해석 레이어는 **시계열 이상탐지 파이프라인의 하류(downstream)**에 위치한다. 즉 정류장별 이상 점수·잔차·정체시간 등 **수치 판정은 이미 시스템이 계산 완료**한 상태에서, LLM은 그 결과 + 부가 컨텍스트(날씨, 요일, 사용자 신고)를 **해석·설명·태깅**하는 역할만 맡는다. 이 경계 설정 자체가 4번 섹션(환각 억제)의 핵심 전제다.
- **[추정]** 정류장 수 ~300개, 5분 폴링 기준이라는 전제는 사용자가 제시한 값을 그대로 사용했다(대전 타슈 실제 정류장 수를 별도 검증하지 않음).

## 1. Claude API 사용 패턴

### 1.1 구조화 출력(JSON 강제)

**[확인]** 두 가지 메커니즘이 있고 병행 가능하다 (`docs/en/build-with-claude/structured-outputs.md` 직접 확인):

| 메커니즘 | 용도 | 핵심 |
|---|---|---|
| `output_config: {format: {type: "json_schema", schema: {...}}}` | 응답 본문 전체를 스키마에 맞는 JSON으로 강제 | **constrained decoding**(그래머 기반 샘플링)으로 문법·필수필드·타입을 구조적으로 보장 — 프롬프트로 "JSON만 출력해" 하는 것과 차원이 다름. 파싱 재시도가 사실상 불필요 |
| `tools=[{..., "strict": true}]` | 도구 호출 파라미터를 스키마에 맞게 강제 | 멀티스텝 흐름(예: 과거 이력 조회 도구 → 최종 판정 도구)에서 마지막 "판정 제출" 도구를 강제 호출시킬 때 유리 |

**JSON 스키마 제약(중요, 설계에 직접 영향):**
- 지원: `object/array/string/integer/number/boolean/null`, `enum`, `const`, `anyOf`, `additionalProperties: false`(필수)
- **미지원: `minimum`/`maximum`/`multipleOf` 같은 숫자 범위 제약, `minLength`/`maxLength`** — 즉 "신뢰도는 0~1 사이 숫자"를 스키마 레벨에서 강제할 수 없다. → 3번 섹션에서 신뢰도를 **숫자가 아닌 enum 버킷**으로 설계한 이유가 여기서 나온다.
- 권장 SDK 경로: Python `client.messages.parse(output_format=PydanticModel)`, TS `zodOutputFormat(ZodSchema)` — 스키마 정의와 파싱 검증을 한 번에 처리.
- **제한사항**: `refusal`(안전 거부) 시 스키마 미준수 가능, `max_tokens` 도달 시 불완전 출력 가능, Citations 기능과 병행 불가.

**단순 1콜 해석(판정만)** → `output_config.format`이 적합. **추가 조회가 필요한 심층 해석**(예: 최근 24시간 추세를 별도 도구로 가져와야 함) → Tool Runner로 루프를 돌리고 마지막에 `strict: true`가 걸린 `submit_verdict` 도구를 `tool_choice`로 강제.

### 1.2 프롬프트 캐싱으로 시스템 프롬프트 비용 절감

**[확인]** (`prompt-caching.md` 및 pricing 페이지 확인)
- 캐싱은 **prefix 매치**: 렌더 순서가 `tools → system → messages`이므로, 판정 기준·JSON 스키마 설명·few-shot 예시가 담긴 시스템 프롬프트를 **고정된 앞부분**에 두고 `cache_control: {type:"ephemeral"}`을 마지막 시스템 블록에 건다.
- 가격 배율: 5분 캐시 쓰기 1.25×, 1시간 캐시 쓰기 2×, **캐시 히트는 0.1×**(기본 입력가 대비). 5분 TTL은 2번 읽으면 손익분기, 1시간 TTL은 3번 읽어야 손익분기.
- **이 워크로드에 특히 유리한 이유**: 정류장별 이상 신호 해석 요청은 하루 수백~수천 건이 같은 시스템 프롬프트(판정 기준, 태깅 카테고리, 스키마)를 공유한다. 시스템 프롬프트가 1,500토큰이라면, 캐싱 없이는 매 요청마다 $ (1,500/1e6)×기본단가를 내지만 캐싱하면 대부분의 요청이 0.1×만 낸다.
- **주의**: 도구 목록/모델을 중간에 바꾸면 캐시 전체가 무효화된다. 판정 기준(시스템 프롬프트)은 하루 중 고정, 변경이 필요하면 배치 처리 사이(예: 매일 자정 배포)에만 바꾼다.
- `usage.cache_read_input_tokens`가 반복 요청에서 0이면 시스템 프롬프트에 타임스탬프/UUID 같은 비결정적 요소가 섞여 있다는 신호 — 반드시 점검 대상.

### 1.3 Batch API vs 실시간 Messages API

**[확인]** (`build-with-claude/batch-processing.md`, pricing 페이지 확인)

| | Messages API (실시간) | Batch API |
|---|---|---|
| 지연 | 초 단위 | 대부분 1시간 이내, 최대 24시간 |
| 가격 | 기본가 | **입력·출력 모두 50% 할인** (예: Sonnet 5 인트로가 기준 입력 $2→$1, 출력 $10→$5) |
| 결과 순서 | 즉시 응답 | **순서 보장 안 됨** — `custom_id`로 키 매칭 필수 |
| 적합 용도 | 고장 의심 판정(신호 발생 직후 대응 필요), 사용자 신고 즉시 분류 | 일일 브리핑 생성, 하루치 이상 이력 재요약, 과거 데이터 재분석/평가(eval) |

**이 파이프라인에서의 구체적 적용:**
- **고장 의심 판정 + 사용자 신고 분류**: 실시간 Messages API 필수(운영자가 즉시 알림을 받아야 함).
- **일일 브리핑**: 브리핑 자체는 하루 1회 생성이라 호출 수가 적어 Batch API의 이득이 크지 않을 수 있음(→ [추정] 이 경우 실시간 API로도 충분). 다만 브리핑 파이프라인이 "정류장별 미니 요약을 N개 병렬 생성 후 롤업"하는 구조라면(예: 그날 이상이 있었던 정류장 30~50곳 각각에 대해 짧은 요약을 먼저 만들고 합치는 2단계 구조), 이 병렬 요약 단계는 Batch API로 돌려 비용을 절반으로 줄이고 지연도 문제 없음(브리핑은 익일 아침 배포이므로 24시간 SLA에 충분히 여유).
- Batch 요청 구성 시 `custom_id`에 `station_id + date`를 넣어 결과를 정류장별로 재조립.

| Claude Sonnet 5 | `claude-sonnet-5` | $2/MTok (2026-08-31까지 인트로가, 이후 $3) | $0.20/MTok | $10/MTok (이후 $15) | 1M 컨텍스트, 코딩·에이전틱에서 이전 Opus급 품질 |

Batch 할인은 위 표의 정확히 50%(예: Sonnet 5 인트로가 배치 입력 $1, 출력 $5).

### 2.2 티어별 배치 제안

**Tier 1 — 실시간 대량 분류 (저지연·저비용): `claude-haiku-4-5`**
- 대상: (a) 이상탐지 시스템이 1차로 걸러낸 **후보 신호**에 대한 1차 트리아지("이게 진짜 이상 후보인가 / 무시해도 되는 변동인가"), (b) **사용자 신고 텍스트 고장 유형 태깅**.
- 근거: 이 작업들은 스키마가 얕고(수 개 enum 필드), 컨텍스트도 짧고(주입된 수치 몇 개 + 신고 원문), 정답이 비교적 명확한 분류 문제다. Haiku 4.5는 200K 컨텍스트로 충분하고, 이 계층의 물량(하루 수백~수천 건)에서 Sonnet 대비 5분의 1 가격 차이가 누적되면 유의미하다. `strict: true` + enum 스키마로 오분류 시 재시도 비용도 억제 가능.
- 효과(effort) 설정: Haiku 4.5는 `effort`/adaptive thinking 파라미터를 걸어도 되지만 이 정도 분류 작업엔 사실상 불필요 — thinking 없이 저지연 우선.

**Tier 2 — 심층 해석(고품질): `claude-sonnet-5`**
- 대상: Tier 1을 통과한(또는 신뢰도가 애매하게 나온) 후보에 대해 **"고장 의심 판정 + 근거 설명 + 권장조치"**를 생성하는 본 작업. 여러 신호(이상 점수, 잔차, 정체시간, 날씨, 요일, 인접 정류장 추세, 최근 유사 사례)를 종합 추론해야 하므로 판단 품질이 중요하다.
- 근거: Sonnet 5는 코딩/에이전틱 작업에서 이전 세대 Opus급 품질에 도달했다고 문서화되어 있고, 가격은 Opus의 40%(인트로가 기준) 수준이다. 이 작업은 "여러 신호를 교차 검증해 하나의 판정으로 수렴"하는 다단계 추론이 필요하므로 Haiku보다 Sonnet이 적합. `effort: "medium"`을 기본값으로 시작해 오탐/미탐률을 보며 `high`로 조정할 것을 권장(운영 초기에는 `high` 시작 후 안정화되면 낮추는 것도 방법).
- **일일 브리핑 생성도 이 티어**: 하루치 이상 이력 + 신고 데이터를 종합해 운영자용 내러티브를 쓰는 작업은 Sonnet 5로 충분하고, 실시간성이 없으므로 위 1.3의 Batch API 적용 대상.

**Tier 3(선택, 에스컬레이션) — `claude-opus-4-8`**
- 대상: Tier 2 판정 신뢰도가 낮게 나왔거나(또는 사람이 "보류" 건을 재검토 요청했을 때), **현장 출동 여부를 결정하는 고비용 액션 직전**의 최종 확인 단계.
- **[추정]** 이 파이프라인 규모(300개 정류장, 5분 폴링)에서는 Tier 3 물량이 하루 한 자릿수 건일 것으로 예상되므로, 상시 운용보다는 "보류" 판정이 반복되거나 운영자가 명시적으로 재해석을 요청한 케이스에 온디맨드로만 호출하는 게 비용 효율적이다. 필수는 아니고 선택적 안전망으로 제안.

```
당신은 대전 공영자전거 타슈의 정류장 이상탐지 결과를 해석하는 운영 보조 시스템입니다.

## 역할과 제약
- 아래 사용자 메시지에 포함된 수치(이상 점수, 잔차, 정체 시간, 날씨, 요일 등)는
  이미 시스템이 계산·검증을 마친 값입니다. 당신은 이 값을 재계산하거나 추정하지 않고,
  주어진 값을 그대로 근거로 삼아 "해석"만 수행합니다.
- 절대로 새로운 수치(구체적인 대여/반납 건수, 시각, 백분율 등)를 만들어내지 마세요.
  설명(reasoning) 텍스트에서 숫자를 언급해야 한다면 반드시 입력에 주어진 값만
  그대로 인용하십시오.
- 입력된 정보만으로 확신 있는 판정을 내리기 어렵다면 반드시 verdict를 "보류"로
  설정하십시오. 애매할 때 "고장_의심"이나 "정상_변동" 중 하나를 억지로 고르지 마세요.
- confidence는 숫자가 아니라 categorical(낮음/중간/높음)입니다. 당신의 주관적
  확신 수준을 나타내는 것이며, 시스템이 계산한 anomaly_score와는 다른 개념입니다.

## 고장 의심 판정 기준 (참고)
- anomaly_score가 임계값을 크게 초과 + 잔차 부호가 지속적으로 한 방향
  (예: 반납은 계속 발생하는데 대여가 0에 가까움) → 거치대 잠금/통신 계열 의심
- 정체 시간이 비정상적으로 길고 날씨/요일로 설명되지 않음 → 물리적 고장 의심
- 날씨(우천/폭염) 또는 요일(주말/공휴일) 요인으로 정상적으로 설명 가능한 변동
  → "정상_변동"
- 주변 정류장도 동시에 유사 패턴 → 개별 정류장 고장보다 지역적 수요 변화 가능성
  (반드시 명시)
```

### 3.2 JSON 스키마 (`output_config.format`)

```json
{
  "type": "object",
  "properties": {
    "station_id": {
      "type": "string",
      "description": "입력으로 주어진 정류장 ID를 그대로 반환 (재입력 검증용)"
    },
    "verdict": {
      "type": "string",
      "enum": ["고장_의심", "정상_변동", "보류"]
    },
    "confidence": {
      "type": "string",
      "enum": ["낮음", "중간", "높음"],
      "description": "숫자가 아닌 정성적 확신도. 시스템의 anomaly_score와 별개."
    },
    "suspected_cause": {
      "type": "string",
      "enum": [
```

## 4. 환각 억제 장치

이 레이어의 신뢰성은 "LLM이 숫자를 만들어내지 않는다"는 보장에 달려 있다. 아래를 층층이 적용한다.

1. **수치는 시스템 주입, LLM은 해석만 — 구조적으로 강제**
   - LLM 출력 스키마에는 **새로운 수치 필드가 없다**(3.2 스키마 참고: `confidence`도 숫자가 아니라 enum). LLM이 숫자를 "생성"할 스키마상의 통로 자체를 없앤 것 — 프롬프트 지시("숫자 만들지 마")보다 구조적 제약이 훨씬 강한 보장이다.
   - `station_id`처럼 입력값을 그대로 반환해야 하는 필드는 시스템이 **사후 대조 검증**(입력값 == 출력값)해서 불일치 시 해당 응답 폐기 후 재시도 — LLM이 엉뚱한 정류장 ID를 "기억"해서 오출력하는 것을 잡아낸다.

2. **`referenced_fields`로 근거 추적성 확보**
   - `reasoning`에서 어떤 입력 필드를 근거로 썼는지 별도 배열로 받는다. 시스템은 이 배열의 필드명이 실제로 입력 JSON에 존재하는지 검증하고, 존재하지 않는 필드명을 참조했다면(=환각) 해당 응답을 무효 처리하거나 낮은 신뢰도로 강등한다.

3. **`reasoning` 텍스트 내 숫자 사후 정합성 검사**
   - 응답 후처리 단계에서 `reasoning` 문자열에서 정규식으로 숫자를 추출하고, 이 숫자들이 입력 JSON에 실제로 존재하는 값(또는 그 반올림)과 일치하는지 대조한다. 불일치하는 숫자가 발견되면: (a) 해당 문장을 UI에 노출하지 않거나, (b) 시스템이 준비한 템플릿 문구로 치환하거나, (c) 전체 응답을 "보류" 등급으로 강등하고 재시도 큐에 넣는다.

4. **JSON 스키마 강제(`output_config.format` / `strict: true`) + `additionalProperties: false`**
   - 자유 텍스트 응답이 아니라 constrained decoding으로 스키마를 강제하므로 "판정 필드가 스키마 밖 자유 서술로 새는" 경로 자체가 차단된다.

5. **"보류" 클래스 강제 조건 — 애플리케이션 레벨 규칙**
   - 아래 중 하나라도 해당하면 **LLM 호출 이전에 시스템이 먼저 `verdict = 보류`로 확정**하고 LLM은 근거 설명 생성에만 관여(즉 LLM이 보류를 "판단"하는 게 아니라, 특정 조건에서는 애초에 판정 선택권을 주지 않음):
     - `anomaly_score`가 임계값에서 ±10% 이내(경계값 근처, [추정] 구체 수치는 실측 후 튜닝)
     - `context` 필드 중 필수 항목(날씨 등)이 null/누락
     - 최근 24시간 내 동일 정류장에 대해 이미 3회 이상 판정이 뒤집힌 이력(진동하는 신호)
   - LLM 자체 판단으로 confidence가 "낮음"으로 나온 경우도 UI/알림 단계에서 "보류"와 동일하게 취급(자동 조치 트리거하지 않고 사람 확인 큐로).

6. **모델 응답 원본 로깅 + 재현성**
   - 모든 판정에 대해 (입력 JSON, 시스템 프롬프트 버전, 모델 ID, 원본 응답)을 저장. 오탐 사후 분석 및 프롬프트 회귀 테스트(같은 입력을 재실행해 판정이 안정적인지)에 사용. `request_id`(`_request_id` 필드)도 함께 저장해 Anthropic 측 문의 시 활용.

7. **구조화 출력의 한계 인지**
   - `refusal`(안전 거부) 시나 `max_tokens` 도달 시 스키마가 깨질 수 있음 — `stop_reason`을 항상 먼저 확인하고, `end_turn`이 아니면 스키마 검증을 건너뛰고 즉시 "보류 + 시스템 오류" 상태로 분류.

## 5. 비용 추정

### 5.1 물량 산정

- **[추정 전제]** 정류장 300개 × 5분 폴링 = 1일 폴링 횟수 = 300 × (24×60/5) = **86,400회/일**.
- **[추정]** 이 86,400건 전부를 LLM에 보내는 것은 설계상 배제한다(6번 방지 장치 참고) — 시계열 이상탐지 시스템이 통계적으로 1차 필터링해 **"이상 후보"만** LLM에 넘기는 구조가 전제.
- **[추정]** 후보 비율: 유사 규모 공유자전거 이상탐지 사례를 참고할 근거는 없어 순수 추정치다. 보수적으로 **전체 폴링의 1~2%**를 이상 후보로 가정(하루 864~1,728건). 이 중 Tier 1(Haiku)이 전부를 처리하고, Tier 2(Sonnet)로 에스컬레이션되는 비율은 **Tier 1 결과의 약 15%**로 가정(864~1,728건의 15% = 130~260건/일).
- 사용자 신고: **[추정]** 하루 30~80건 (대전시 규모 공영자전거 민원 접수량에 대한 근거 데이터 없음 — 순수 가정치).
- 일일 브리핑: 1회/일.

### 5.2 토큰 규모 (추정)

| 작업 | 입력(캐시 제외 실질) | 출력 | 비고 |
|---|---|---|---|
| Tier 1 트리아지 | ~800 tok(신호+컨텍스트, 시스템프롬프트는 캐시) | ~150 tok | Haiku |
| Tier 1 신고 분류 | ~300 tok | ~100 tok | Haiku |
| Tier 2 심층 판정 | ~2,000 tok(신호+주변정류장+신고이력, 시스템프롬프트 캐시) | ~400 tok | Sonnet 5 |
| 일일 브리핑 | ~8,000 tok(하루치 요약 롤업) | ~2,000 tok | Sonnet 5, 실시간 1콜 가정 |

### 5.3 일일 비용 계산 (인트로 가격, [확인] 단가 × [추정] 물량)

**Tier 1 — 이상 신호 트리아지 (Haiku 4.5, 864~1,728건/일 가정)**
- 건당: (800/1e6)×$1 + (150/1e6)×$5 = $0.0008 + $0.00075 ≈ **$0.00155**
- 일일: 864~1,728건 × $0.00155 ≈ **$1.34 ~ $2.68/일**

**Tier 1 — 사용자 신고 분류 (Haiku 4.5, 30~80건/일)**
- 건당: (300/1e6)×$1 + (100/1e6)×$5 = $0.0003+$0.0005 = **$0.0008**
- 일일: 30~80건 × $0.0008 ≈ **$0.024 ~ $0.064/일** (무시 가능한 수준)

**Tier 2 — 심층 판정 (Sonnet 5 인트로가, 130~260건/일)**
- 건당: (2,000/1e6)×$2 + (400/1e6)×$10 = $0.004+$0.004 = **$0.008**
- 일일: 130~260건 × $0.008 ≈ **$1.04 ~ $2.08/일**

**일일 브리핑 (Sonnet 5, 1건)**
- (8,000/1e6)×$2 + (2,000/1e6)×$10 = $0.016+$0.02 = **$0.036/일**
- (Batch API로 하부 요약 단계를 돌린다면 이 항목은 절반 수준으로 더 낮아짐)

**합계**
- 하루: 약 **$2.4 ~ $4.9/일**
- 월(30일): 약 **$72 ~ $147/월**
- **캐싱 반영 시 추가 절감**: 위 계산은 시스템 프롬프트를 "캐시 제외 실질 입력"으로 이미 분리했으므로 시스템 프롬프트 자체의 반복 비용은 위 수치에 거의 반영 안 됨(캐시 히트 0.1×로 사실상 무시 가능한 수준까지 떨어짐). 즉 위 추정치가 이미 캐싱을 전제로 한 "실질 비용"에 가깝다.

### 5.4 비교: 필터링 없이 전량을 LLM에 보낼 경우 (설계 반례로 제시)

- 86,400건/일 × Haiku 트리아지 단가($0.00155) ≈ **$134/일 ≈ $4,020/월** — Tier 1만으로도 위 필터링 설계 대비 약 **30~50배** 비용.
- 이 비교가 시사하는 설계 원칙: **"이상 후보만 LLM에 넘긴다"는 전제(사용자가 과제에서 명시)가 비용을 결정하는 가장 큰 변수**다. 시계열 이상탐지 모델의 1차 필터 재현율(recall)을 충분히 높게 유지하면서 후보 비율을 낮게 유지하는 것이 LLM 레이어 비용 설계의 핵심 레버.

### 5.5 확장성 여유

- 300개 정류장 규모에서는 위 비용이 무료 크레딧 수준~소규모 SaaS 구독료 수준으로, Anthropic API의 표준 레이트리밋(Start/Build/Scale 티어)로 충분히 감당 가능한 물량이다(**[추정]**, 구체 RPM/TPM 한도는 조직의 실제 사용 티어에 따라 달라지므로 별도 확인 필요 — `platform.claude.com/docs/en/api/rate-limits`).
- 정류장 수가 10배(3,000개) 늘어나도 위 산식이 선형 스케일하며 여전히 월 $1,000 미만 수준으로 예상된다.
