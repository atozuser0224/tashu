# 1부. 기획

> **문서 상태: 보존된 초안.** 현재 제품 기준은 [`docs/v2/01_PRODUCT_MVP.md`](../v2/01_PRODUCT_MVP.md)다.

## 1.1 문제 정의와 근거

### 1.1.1 현상

대전 타슈 이용자는 앱에서 "이용가능"으로 표시된 자전거·정류장을 찾아갔지만 실제로는 잠금장치가 열리지 않거나, 자전거가 파손돼 있거나, 배터리(전기자전거의 경우 태양광 충전 패널이 낙엽·쓰레기 등으로 가려져 방전된 사례가 확인됨)가 소진된 상태를 마주치는 경우가 있다. 반대 방향의 오표기도 확인된다 — 정상적으로 반납했음에도 앱에는 "미반납"으로 남아 요금이 자동 청구되는 민원 사례가 존재한다. 두 현상은 방향만 반대일 뿐 **동일한 근본 원인**, 즉 "서버가 표시하는 상태 값과 현장 자전거·거치대의 실제 상태 간 동기화 지연/누락"을 가리킨다는 것이 이번 기획의 핵심 전제다.

### 1.1.2 타슈 API가 "주는 것"과 "주지 않는 것" — 이 기획의 출발점

타슈 오픈API(`bikeapp.tashu.or.kr:50041/v1/openapi/station`)를 실측·교차검증한 결과는 다음과 같다.

**제공되는 것 (확인)**
- 정류소 단위 식별자(`id`), 명칭, 주소, 좌표(`x_pos`=위도, `y_pos`=경도 — 필드명과 실제 의미가 반대이므로 개발 시 주의), 그리고 **정류소별 현재 대여가능 자전거 수(`parking_count`)** — 딱 6개 필드로 구성된 정류소 단위 집계값 하나뿐이다. 이 6필드 구성은 언어·플랫폼이 서로 다른 6개 이상의 독립 서드파티 프로젝트에서 일관되게 확인되어 신뢰도가 높다.

**제공되지 않는 것 (확인 — 부재가 여러 소스에서 교차 확인됨)**
- **개별 자전거 ID·상태**: 실시간 API에는 개별 자전거를 식별할 방법이 전혀 없다. (다만 자전거 고유번호 자체는 시스템에 존재하며, 연 1회 갱신되는 대여이력 CSV에만 담겨 있다.)
- **고장/파손 여부 필드**: 실시간 API, 대여이력 CSV 어디에도 존재하지 않는다.
- **거치대 총 용량(도크 수)**: `parking_count`(현재 대수)만 있고 최대 수용량이 없어 "꽉 찼는지"를 API만으로 판단할 수 없다.
- **실시간 대여/반납 이벤트(대여 시도 실패 포함)**: 존재하지 않는다. 이력은 연 1회 배치 CSV로만 내려온다.

참고로 국제 표준 GBFS(General Bikeshare Feed Specification)에는 `num_vehicles_disabled`, `num_docks_disabled` 같은 "고장/비활성 대수"를 분리해 노출하는 표준 필드가 정의되어 있다. 타슈가 이 필드를 공개하지 않는 것은 GBFS를 따르지 않거나, 따르더라도 해당 옵션 필드를 의도적으로 비공개했을 가능성을 보여준다(추정) — 즉 "기술적으로 불가능해서"가 아니라 "공개 정책상 제공하지 않는" 영역일 가능성이 있다는 뜻이며, 이는 우리가 값을 직접 계산·추정해서 메워야 하는 이유이자 동시에 향후 운영기관 제휴 시 협상 여지가 되는 지점이다.

**이 프로젝트의 존재 이유는 정확히 이 공백에서 나온다.** 우리는 "정류소별 총량"이라는 저해상도 신호만으로, 개별 자전거·거치대 단위의 "실질 이용가능 여부"를 통계적으로 추정하고, 사람(이용자·정비사)의 확인으로 이를 검증·보정하는 시스템을 만든다.

### 1.1.3 근본 원인 구조

| 계층 | 근본 원인 | 근거 |
|---|---|---|
| 데이터 소스 | 정류소 집계값만 노출, 자전거 단위 상태 필드 부재 | API 실측 (R1) |
| 시스템 | 서버-현장기기 실시간 동기화 지연/누락 | 미반납 오표기 민원 사례 (R2) |
| 업계 공통성 | 조사된 타 지자체 공영자전거(따릉이, 피프틴, 어울링, 타조 등) 전반에서 동일 패턴 확인 | 다수 2차 출처 비교 (R2) — 즉 대전만의 문제가 아니라 구조적 문제이며, 우리 솔루션은 타 지자체로 확장 가능한 도메인이다 |

## 1.2 목표와 비목표(Non-goals)

### 1.2.1 제약 조건 — "공식 앱을 직접 수정할 수 없다"

우리는 타슈 앱의 소스코드나 서버 인프라에 접근 권한이 없다. 운영기관과의 공식 제휴(MOU) 없이는 공식 앱의 표시 로직 자체를 바꿀 방법이 없다. 이 제약은 R2에서 정리한 세 가지 배포 형태 중 우리가 초기에 취할 수 있는 형태를 사실상 결정한다.

| 형태 | 설명 | 초기 채택 가능성 |
|---|---|---|
| ① 별도 보정 앱/웹(비공식) | 오픈API 재가공 + 크라우드소싱 신고 오버레이를 독립 서비스로 배포 | **채택** — 운영기관 승인 없이 즉시 시작 가능 |
| ② 운영기관 제휴(MOU) | 우리 신고 데이터를 공식 정비 티켓 시스템에 직결 | 목표(비전)로 설정하되 초기 범위에서는 제외. 실증 트랙레코드 확보 후 제안 |
| ③ 공익 대시보드(통계 공개) | 개인 대상 서비스가 아닌 통계·투명성 자료 공개 | ①과 병행 가능한 보조 산출물 |

따라서 이번 기획의 산출물은 **"타슈 공식 앱의 대체·보정 정보를 제공하는 독립 서비스(웹/앱)"**이며, 모든 화면에 "타슈 공식 서비스 아님, 추정 정보"임을 명확히 고지하는 것을 설계 원칙으로 못박는다(운영기관 신용 훼손 소지, 이용자 오인 리스크 방지 — R2 4장).

### 1.2.2 목표(Goals)

1. 정류소 집계 데이터 + 이용자/정비 신고를 결합해, 공식 앱보다 **신뢰도 높은 "실질 이용가능 여부" 추정치**를 정류소(및 가능한 범위 내 개별 자전거) 단위로 제공한다.
2. 이상탐지 AI가 "의심 정류소"를 선제적으로 좁히고, LLM이 신호를 사람이 이해할 수 있는 판정·근거로 변환한다.
3. 신고(체크인) → 확인 → 해제로 이어지는 상태 추적 시스템을 통해, 고장 정보가 방치되지 않고 "언제 해결됐는지"까지 추적 가능하게 한다.
4. 초기엔 비공식 독립 서비스로 시작해, 데이터·트랙레코드를 축적한 뒤 운영기관 제휴(공식 채널 편입)를 협상할 근거를 마련한다.

### 1.2.3 비목표(Non-goals)

- **공식 타슈 앱/서버를 직접 수정하지 않는다.** 우리 시스템의 판정은 공식 데이터를 대체하는 것이 아니라 "참고용 보정 정보"로 명시한다.
- **개별 자전거의 실시간 GPS 위치를 자체 추적하지 않는다.** API가 이 데이터를 제공하지 않으며, 우회 수집(무단 스크레이핑 등)은 이용약관 위반 소지가 있어 시도하지 않는다(R2 4장).
- **초기 단계에서 정비사 출동·수리 자체를 대행하지 않는다.** 정비 체크인은 운영기관 인력이 주체이며, 우리는 그 정보를 수집·표시하는 역할에 한정한다(제휴 전까지는 정비 체크인 데이터 자체를 확보할 수 없다는 뜻이기도 하다 — 1.6 로드맵 참고).
- **결제/과금 시스템에는 관여하지 않는다.** 미반납 오표기로 인한 요금 분쟁은 문제 인지·신고 채널까지만 지원하고, 정산은 공식 채널로 안내한다.
- **전국 단위 동시 확장은 초기 범위에서 제외한다.** 구조적으로 타 지자체 확장이 가능함은 확인했으나(1.1.3), MVP는 대전 타슈 단일 시스템에 집중한다.

## 1.3 페르소나 3종과 핵심 시나리오

### 페르소나 A — 이용자 (일반 시민 라이더)

- **프로필**: 출퇴근·근거리 이동에 타슈를 쓰는 20~40대. 앱의 "이용가능" 표시를 믿고 정류소로 이동한다.
- **핵심 페인포인트**: 정류소에 도착했더니 표시된 대수만큼 자전거가 없거나, 있어도 잠금이 안 풀리거나 파손돼 있는 "허탕" 경험.
- **핵심 시나리오**:
  1. 앱(우리 서비스)에서 목적지 인근 정류소의 "보정된 실질 이용가능 대수"와 신뢰도 뱃지를 확인하고 이동한다.
  2. 현장에서 실제로 문제(잠금불가/파손/배터리없음 등)를 겪으면, QR 스캔 또는 정류소·자전거 번호 입력 + 문제유형 선택으로 **1분 내 신고(이용자 체크인)**를 남긴다.
  3. 자신의 신고가 시스템에서 어떻게 처리되는지("확인 중" → "고장 확정" → "해제") 상태를 추적할 수 있다.

### 페르소나 B — 정비·운영자 (타슈 현장 정비 인력 또는 우리 측 파일럿 협력 인력)

- **프로필**: 정류소를 순회하며 자전거 상태를 점검·수리하는 인력. 초기(제휴 전)에는 우리 조직 내부 파일럿 인력, 제휴 이후에는 운영기관 정비팀으로 확장.
- **핵심 페인포인트**: 어디부터 점검해야 할지에 대한 우선순위 정보 부재 — 순회는 고정 루트/랜덤이며 "지금 이상 신호가 강한 곳"을 콕 짚어주는 도구가 없다.
- **핵심 시나리오**:
  1. 대시보드에서 AI 신뢰도 점수(높음/중간/낮음) 및 이용자 신고가 겹치는 "우선 점검 대상" 정류소 목록을 확인한다.
  2. 현장 도착 시 GPS 기반 도착 체크인 → 개별 자전거 QR 스캔 → 상태(정상/경미결함-현장수리/중대결함-회수) 기록.
  3. 조치 완료 시 "정비 체크인" 제출 → 해당 정류소·자전거 상태가 `resolved`로 전환되고, 동시에 이 데이터가 AI 모델의 정답(ground truth)로 축적된다.

### 페르소나 C — 서비스 관리자 (우리 팀 운영자)

- **프로필**: 이상탐지 모델·LLM 판정 파이프라인·신고 데이터를 종합 모니터링하고, 오탐/미탐을 조정하며, 향후 운영기관 제휴를 추진하는 내부 담당자.
- **핵심 페인포인트**: 라벨(확정된 고장 사례) 자체가 희소한 콜드스타트 상황에서 모델 신뢰도를 어떻게 검증·개선할지, 그리고 오탐이 누적되면 이용자·정비 인력의 신뢰를 잃는 문제.
- **핵심 시나리오**:
  1. 매일 아침 **LLM이 생성한 운영 브리핑**(전일 이상 정류소 목록, 신고 처리 현황, 신뢰도 추이)을 확인한다.
  2. 오탐/미탐 사례를 "정탐/오탐" 클릭으로 라벨링해 모델 재학습 파이프라인에 피드백한다.
  3. 정류소별 Precision@k, 리드타임 등 KPI 대시보드를 보며 임계값·가중치를 조정하고, 축적된 트랙레코드를 근거로 운영기관 제휴 제안서를 준비한다.

## 1.4 핵심 기능 정의

### 1.4.1 실제 이용가능 대수 보정 표시

- **입력**: 타슈 실시간 정류소 API의 `parking_count`(집계값) + 이상탐지 신뢰도 점수 + 최근 이용자/정비 체크인 이력.
- **출력**: 정류소 카드에 "공식 표시: N대" 옆에 **"보정 신뢰도: 높음/중간/낮음"** 뱃지를 병기한다. 개별 자전거 단위의 확정적 판정("이 자전거는 고장")은 API 구조상 불가능하므로, 정류소 단위의 "이 정류소 표시치를 얼마나 믿을 수 있는가"로 설계한다.
- **원칙**: 절대 "실제 이용가능 대수는 X대"라고 새로운 숫자를 확정 표기하지 않는다 — 우리가 가진 것은 통계적 신뢰도이지 실측치가 아니기 때문이다. UI에 "타슈 공식 데이터 아님, 추정치" 고지를 상시 노출한다(1.2.1 원칙).

### 1.4.2 고장 의심 자동 감지 (시계열 AI)

정류소별 `parking_count` 시계열만으로 도출 가능한 신호는 확정 판정이 아니라 "의심 스코어"에 머무른다는 것이 리서치의 핵심 결론이다. 이에 따라 다음 하이브리드 구조를 채택한다.

- **축 A (예측 잔차 기반)**: LightGBM 잔차 회귀(정류소·시간·요일·날씨·인접정류소 피처를 활용한 전역 pooled 모델)로 기대 점유율을 예측하고, 로버스트 z-score + 지속성 필터(CUSUM)로 단발성 튐(재배치 트럭 등)을 걸러낸 이상 신호만 남긴다.
- **축 B (정체 탐지 규칙)**: 정류소·시간대별 "정체 지속시간" 분포에서 상위 percentile을 적응형 임계값으로 사용. 만차인데 회전이 없는 경우(반납 불가 의심)에 특히 높은 가중치를 둔다.
- **베이스라인**: STL 분해 + 잔차, 신규 정류소 콜드스타트 대응.
- **한계 고지**: 정상 반납했는데 미반납으로 표기되는 유형(1.1.1)은 정류소 집계 데이터만으로는 원리적으로 감지 불가능하다 — 이 유형은 전적으로 이용자 체크인에 의존한다. 자동 감지가 커버하는 것은 "재고 이상"류이지 전체 고장 유형이 아님을 설계·문서 전반에서 명확히 한다.
- **저장/파이프라인**: TimescaleDB(정류소 메타·신고 라벨·날씨 조인 필요성 때문에 InfluxDB보다 우위로 판단), darts를 실험 표준 프레임워크로, river를 배치 재학습 사이 갭을 메우는 온라인 보조 탐지기로 사용.

### 1.4.3 체크인 시스템 (신고 → 확인 → 해제)

공통 상태 모델: `normal(정상) → suspected(의심) → confirmed(고장확정) → resolved(수리완료) → normal`, 분기로 `dismissed(오신고 기각)` / `auto-cleared(자동해제)`.

**(a) 이용자 신고 체크인**
`normal → (신고 1건) pending → (임계치 N건 도달, 동일 IP/기기 중복 배제) suspected → (교차검증) confirmed | dismissed → resolved → normal`
QR 스캔 또는 정류소명+자전거번호 수동입력, 문제유형 선택, 사진 첨부(선택, 신뢰도 가중치용). 동일 대상에 대한 반박 체크인("정상 사용했음")은 신뢰도를 낮추는 방향으로 반영한다.

**(b) 정비 체크인**
`flagged(정비대상 지정) → assigned → arrived(GPS 도착 체크인) → inspecting → returned_to_service | withdrawn | parts_pending`
가장 신뢰도 높은 데이터 소스로, 다른 두 체크인의 검증 기준(ground truth) 역할을 겸한다. **단, 이 체크인은 운영기관 소속 정비 인력 접근이 있어야 성립하므로, 제휴(1.2.1의 형태 ②) 성사 전까지는 제한적 파일럿 규모로만 운용 가능**하다는 점을 로드맵에 명시한다(1.6).

**(c) AI 자동 체크인**
`normal → (이상탐지 스코어>임계치) ai_flagged → (교차검증 성공) suspected/confirmed → (수리 or 정상 순환 재개 관측) resolved/auto-cleared → normal`
AI 단독 판정은 **절대 "고장 확정"으로 사용자에게 노출하지 않고 "확인 필요" 수준의 낮은 신뢰도 뱃지**로만 표시한다. (a) 또는 (b)와 교차검증돼야 `confirmed`로 승격한다.

**LLM 해석 레이어의 역할**: 시계열 모델이 산출한 수치(이상 점수, 잔차, 정체시간 등)는 LLM 호출 이전에 이미 계산이 끝난 값이며, LLM은 이 값을 재계산하지 않고 종합·설명·태깅만 수행하도록 역할을 엄격히 제한한다. 구조화 출력(JSON 스키마 강제)으로 `confidence`를 숫자가 아닌 enum(낮음/중간/높음)으로 설계해 LLM이 새로운 수치를 만들어낼 통로 자체를 차단하고, `referenced_fields`(근거로 참조한 입력 필드 목록)를 사후 대조 검증해 환각을 걸러낸다. 티어링은 사용량·비용 균형을 고려해 1차 트리아지·신고 분류는 경량 모델(Haiku급), 종합 판정·근거 생성은 상위 모델(Sonnet급)로 나누고, 애매하거나 반복적으로 판정이 뒤집히는 사례만 최상위 모델(Opus급)로 온디맨드 에스컬레이션한다.

### 1.4.4 운영 브리핑

전일 이상 정류소 목록, 체크인 처리 현황(신규/진행중/해결), 신뢰도 추이를 종합해 LLM이 매일 1회 서비스 관리자용 내러티브 브리핑을 생성한다. 정류소별 미니 요약을 먼저 생성한 뒤 롤업하는 2단계 구조를 채택하면, 실시간성이 필요 없는 이 하위 요약 단계는 배치 처리로 비용을 절감할 수 있다(구체적 배치/실시간 배분은 실제 운영 물량 확정 후 결정 — 검증 필요).

## 1.5 성공 지표(KPI)

| 지표 | 정의 | 측정 방법 | 비고 |
|---|---|---|---|
| **감지 정밀도 (Precision@k)** | 하루 상위 k건(예: 신뢰도 상위 N건) 알람 중 실제 신고/정비 체크인과 매칭되는 비율 | 알람 로그 vs 체크인(a)(b) 매칭 배치 집계 | k는 운영팀이 감당 가능한 알람 수로 설정 |
| **재현율 (Recall)** | 확보된 확정 고장 사례(체크인 b, 또는 (a)+(b) 교차검증 confirmed) 중 모델이 사전/근접 시점에 알람을 울린 비율 | 확정 라벨 대비 알람 이력 대조 | 초기엔 라벨 자체가 희소하므로 약라벨링(신고 시각 ± N시간 윈도우) 필요 |
| **리드타임 (Lead time)** | 모델 알람 시각과 실제 신고 접수 시각의 차이(음수=선제 탐지 성공) | 알람 타임스탬프 vs 신고 타임스탬프 | 이상탐지의 실질 가치를 보여주는 핵심 지표 |
| **신고→해결 리드타임** | 이용자 체크인 접수(pending) 시점부터 `resolved` 전환까지 걸린 시간의 중앙값/P90 | 체크인 상태 전이 타임스탬프 로그 | 정비 체크인(b) 연동 전에는 "해결"을 정비 확인 없이 자동 순환 재개(auto-cleared)로만 근사할 수밖에 없어 정밀도가 떨어짐 — 제휴 후 지표 신뢰도 향상 예상 |
| **허탕 조우 감소율** | 이용자가 "이용가능"으로 표시된 자전거/정류소를 찾아갔으나 실제 이용 불가였던 경험의 비율 변화 | 베이스라인 부재 — 파일럿 단계에서 자체 설문/체크인 데이터("다녀왔는데 없음·고장" 유형) 표본 조사로 초기값 설정 후 추적 | **검증 필요**: 현재 이 값을 뒷받침할 실측 베이스라인 데이터가 없음. 초기 몇 개월은 지표 확립 자체가 과제 |
| **알람 예산 대비 오탐률** | 하루 알람 건수 상한 제약 하에서의 정탐/오탐 비율 | 운영자의 "정탐/오탐" 피드백 클릭 누적(human-in-the-loop) | precision만 보면 임계값을 극단적으로 올려 알람 0건으로 만들 수 있으므로 알람량과 함께 봐야 함 |
| **캘리브레이션 품질** | AI 신뢰도 등급(높음/중간/낮음)과 실제 확정 비율의 일치도 | Platt scaling 등으로 confidence를 확률처럼 보정 후 신고 데이터로 검증 | 오탐이 신뢰 손상으로 이어지는 것을 조기에 잡기 위한 선행지표 |

배포 전 검증은 **섀도 모드**(신규 모델/임계값을 실제 알람 없이 로그만 쌓아 비교)로 시작하고, 안정화 후 실제 알람 발송으로 전환하는 것을 원칙으로 한다.

## 1.6 단계별 로드맵 (M1~M4)

기간은 리서치에서 직접 제시된 값이 아니라 팀 구성·API 승인 소요기간 등 외부 변수에 좌우되므로 추정치이며, 특히 M1의 API 토큰 승인 소요기간은 **검증 필요**(공지사항에 구체적 소요기간이 명시돼 있지 않고, 실제 서드파티 개발자들이 이를 병목으로 지목한 사례가 확인됨).

| 단계 | 목표 | 주요 산출물 | 기간(추정) |
|---|---|---|---|
| **M1. 데이터 기반 구축** | API 접근 확보 + 수집 파이프라인 | 타슈 오픈API 토큰 신청(승인 대기 리스크 있음, 병행해 data.go.kr `대전광역시_타슈정보` 자동승인 경로를 백업으로 확보 — 단 두 경로가 동일 데이터인지는 미확인이므로 검증 필요), 폴링 수집기, TimescaleDB 적재, 정류소 메타(좌표) 매핑, 최소 대시보드(공식 표시치 그대로 노출) | 4~6주 (API 승인 대기 기간 별도) |
| **M2. 이상탐지 + 이용자 체크인 MVP** | 자동 감지 1차 버전 + 신고 채널 오픈 | STL 베이스라인/정체탐지 규칙(축 B) 우선 배포, 이용자 체크인(a) 신고 UI/API, 독립 웹·앱(형태 ①)으로 공개, "비공식·추정치" 고지 UI 반영 | 6~8주 |
| **M3. LightGBM 고도화 + LLM 해석 레이어 + AI 자동체크인** | 판정 품질 향상, 신뢰도 등급화, 운영 브리핑 | 축 A(LightGBM 잔차 모델) 투입, 하이브리드 스코어 결합, LLM 구조화 출력 파이프라인(Haiku 트리아지 + Sonnet 판정) 연동, `ai_flagged` 자동체크인(c), 일일 운영 브리핑, KPI 대시보드(1.5) | 6~10주 |
| **M4. 트랙레코드 기반 제휴 추진 + 정비 체크인 확장** | 운영기관 제휴 제안, 정비 체크인(b) 정식 도입 | M1~M3에서 축적된 Precision/Recall/리드타임 실증 데이터로 제휴 제안서 작성, (성사 시) 정비팀 체크인 연동, 공익 대시보드(형태 ③) 병행 공개 검토 | 제휴 협상 소요기간에 좌우 — 기간 산정 불가(검증 필요) |

정비 체크인(b)은 구조적으로 M4(제휴 성사) 이전에는 제한적 파일럿 규모로만 존재할 수밖에 없다는 점을 로드맵에 명시적으로 반영했다 — 이는 1.4.3에서 밝힌 제약과 일치한다.

| **규모 추정치 불일치** | 정류소 규모가 리서치 내에서 R3는 "약 1,000개소, 2026년 1,500개소 확대"로, R4의 비용 산정 전제는 "300개소"로 서로 다르게 나타남 | **검증 필요** — M1에서 실제 API 응답으로 정류소 수를 직접 확인하고, 이에 맞춰 인프라 용량·LLM 비용 추정치를 재계산해야 함. 현재 문서의 비용 추정(월 대략 수만~십수만 원대)은 300개소 가정 기준이며 실제 규모가 크면 비례해 증가할 수 있음 |
| **법적/신뢰 리스크 (재배포 오인)** | 우리가 산출한 "고장 의심" 마킹이 공식 데이터처럼 오인되면 이용자 혼란 또는 운영기관 신용 훼손 주장 소지 | UI 전반에 "타슈 공식 서비스 아님, 추정 정보" 고지를 상시 노출(디자인 요소가 아니라 정책으로 못박음), 우회 스크레이핑 없이 정식 토큰 경로만 사용 |
