# 2부. 디자인

> **문서 상태: 보존된 초안.** 이 문서의 보정 대수와 상태 설계는 사용하지 않는다. 현재 기준은 [`docs/v2/01_PRODUCT_MVP.md`](../v2/01_PRODUCT_MVP.md)와 [`contracts/invariants.md`](../v2/contracts/invariants.md)다.

1부에서 확인했듯 타슈 오픈API는 정류장 단위 집계값(`parking_count`)만 제공하며, 개별 자전거 ID·고장 여부·거치대 총 용량 필드는 실시간으로 노출되지 않는다. 이는 디자인 전 영역에 걸친 제약 조건이다 — "이 자전거가 고장났다"를 화면에 확정적으로 표시할 수 있는 데이터 소스가 원천적으로 없기 때문에, 모든 화면은 **추정(AI)과 확인(사람)을 시각적으로 분리**하는 것을 최우선 원칙으로 삼는다. 이 원칙은 1부에서 도출한 3종 체크인(이용자/정비사/시스템 자동)과 신뢰도 스코어링 구조를 그대로 UI 레이어로 옮기는 작업이기도 하다.

---

## 1. 정보 구조(IA)와 사용자 플로우

세 개의 별도 인터페이스로 나눈다. 각각 사용 맥락(이동 중 스마트폰 / 현장 작업 중 스마트폰 또는 태블릿 / 사무실 데스크톱)이 다르기 때문에 하나의 반응형 화면으로 통합하지 않는다.

### 1-1. 이용자 앱 (지도 중심)

```
홈(지도 뷰) ─┬─ 정류장 상세 ─┬─ 고장 신고/체크인 플로우 ─ 제출 완료
             │               └─ (신고 이력 타임라인)
             ├─ 내 신고 내역
             └─ 마이페이지/설정 (알림 설정, 신뢰 등급 등)
```

**플로우**: 지도에서 배지를 확인 → 이상 신호가 있는 정류장을 탭 → 정류장 상세에서 AI 판정 근거를 확인 → (실제로 문제를 발견했다면) 신고 플로우 진입 → 제출 → 이후 "내 신고 내역"에서 상태 변화(신고접수 → 정비중 → 검증완료)를 추적.

핵심 설계 결정: **QR 스캔을 1차 진입점**으로 삼는다. 타슈 시즌2 자전거는 이미 QR 기반 대여/반납 체계를 쓰고 있으므로(1부 R2), 신고 대상 특정 시 QR을 우선 활용하고 스캔이 어려운 경우에만 정류장명+자전거번호 수동 입력으로 폴백한다.

### 1-2. 정비사 모드

```
오늘의 점검 리스트 ─ 정류장 도착 체크인(GPS) ─┬─ 자전거별 점검(QR스캔→상태선택→조치입력→완료)
                                             └─ 정류장 완료 처리 → 다음 정류장
과거 이력 조회
```

**전제 조건 명시**: 이 모드는 1부에서 정리한 배포형태 옵션 중 **② 운영기관 제휴가 성사된 이후에만 실질적으로 작동**한다. 우리가 정비 인력을 직접 통제할 수 없는 한, 정비사 체크인은 "설계는 되어 있으나 데이터가 유입되지 않는" 화면일 수 있다. 이 제약을 감안해 정비사 모드는 화면·데이터 모델을 먼저 정의해두되, 실제 착수 우선순위는 6장의 제작 순서에서 가장 뒤로 둔다.

### 1-3. 운영 대시보드 (웹, 데스크톱 우선)

```
실시간 알림 피드 ─ 정류장 선택 ─ 통합 타임라인(AI신호+신고+정비이력) ─ 조치(정비사 배정/반려)
LLM 일일 브리핑
오탐 피드백 검수 큐
```

**플로우**: 알림 피드(신뢰도순 정렬)를 모니터링 → 이상 신호 클릭 → 상세에서 판정 근거(입력값·`reasoning`·`referenced_fields`) 확인 → 정비사 배정 또는 신고 반려 → 매일 아침 LLM 브리핑으로 전날 요약 확인 → 틈틈이 오탐/정탐 라벨링(1부 R3의 human-in-the-loop 피드백 루프).

## 2. 핵심 화면 명세

### 2-1. 지도 뷰 (이용자 앱, 기본 진입 화면)

**목적**: 대전 전역 정류장의 "실제로 믿을 수 있는" 이용가능 대수를 한눈에 보여준다.

**구성요소**
```
┌───────────────────────────────────┐
│ 🔍 정류장 검색      필터: 전체 ▾     │
├───────────────────────────────────┤
│                                   │
│         [ 지도 영역 ]              │
│     ●7      ●2⚠      ●0          │
│         ●5!      ●3               │
│                                   │
├───────────────────────────────────┤
│ ▲ 가까운 정류장                    │
│ 둔산동 샤크존        2대 ⚠  120m   │
│ 탄방동 한사랑병원     0대     340m  │
│ 시청역 8번출구        5대     510m  │
└───────────────────────────────────┘
```

- 마커 배지 숫자 = **"보정된 실제 이용가능 대수"**: API `parking_count` 원본값에서, 시스템이 고장의심(AI)으로 판정한 대수를 차감해 표시. 배지 옆 `⚠` 아이콘은 AI 보정이 적용됐음을 뜻하며, 탭하면 원본값과 보정값을 나란히 비교해 보여준다.
- **한계 고지**: API는 개별 자전거 단위 상태를 주지 않으므로, 이 "보정값"은 정류장 단위 이상탐지 결과에 근거한 추정치다. 개별 자전거 고장을 자전거 단위로 차감하는 것이 아니라, 통계적으로 "정류장 재고 중 약 N대는 정상 순환에서 이탈한 것으로 보인다"는 근사치임을 상세 화면에서 명확히 밝힌다.
- 필터: 전체 / 이용가능만 / 고장의심 표시 / 최근 신고 있음

**상태**
- 로딩: 스켈레톤 마커
- 정상: 최신 데이터 반영
- 데이터 지연/결측: 폴링 실패(`is_missing` 플래그, 1부 R3)가 감지된 정류장은 배지를 회색조로 낮추고 "정보 지연"으로 표기 — API 장애를 고장으로 오인시키지 않기 위한 핵심 방어장치
- 빈 결과: 신고/이상 없는 지역은 필터 적용 시 "이 조건에 해당하는 정류장 없음" 안내

### 2-2. 정류장 상세

**목적**: 해당 정류장의 집계 데이터와 AI 판정 근거를, 사람이 읽을 수 있는 문장으로 노출한다.

```
┌───────────────────────────────────┐
│ ← 탄방동 한사랑병원                 │
│ 대전 서구 탄방동 730                │
├───────────────────────────────────┤
│ 원본(API) 대수      0대            │
│ 보정된 이용가능      0대            │
├───────────────────────────────────┤
│ ⚠ AI 추정 · 신뢰도 중간             │
│ "이 정류장은 지난 38분간 대여/반납   │
│  변동이 없었고, 인근 정류장은 정상   │
│  순환 중입니다. 날씨·요일 요인으로는│
│  설명되지 않아 거치대 잠금 오류가    │
│  의심됩니다."                      │
│ 근거: 정체시간, 인근정류장 대비      │
│ [정확한가요?  정확해요 / 아니에요]  │
├───────────────────────────────────┤
│ 신고 이력                          │
│ 07/21 13:10  잠김불가 신고 1건      │
├───────────────────────────────────┤
│      [ 여기서 문제 신고하기 ]        │
└───────────────────────────────────┘
```

- AI 판정 근거 카드는 1부 R4의 LLM 출력 스키마(`verdict`, `confidence`, `suspected_cause`, `reasoning`, `referenced_fields`)를 그대로 UI에 매핑한 것이다. **`reasoning` 문장에 등장하는 사실은 반드시 `referenced_fields`로 추적 가능한 값만 표시**하고, 화면에 새로운 숫자(정확한 시각·건수 등)를 노출할 때는 백엔드가 이미 검증한 값만 쓴다.
- **개별 자전거 표시의 한계**: API가 자전거 단위 ID를 주지 않으므로, 정류장 상세에서 "이 자전거가 고장"이라고 개별 특정할 수 있는 경우는 **이용자가 신고 시 자전거 번호를 직접 입력·스캔한 건에 한해서만**이다. 시스템 자동탐지(AI)는 정류장 집계 단위로만 이상을 표시하며, 자전거별 리스트가 아니라 "정류장 이상 신호" 카드 한 장으로만 노출한다. 이 구분을 화면 문구에서 흐리지 않는다.
- 거치대 총 용량(도크 수) 필드는 실시간 API에 없다(1부 R1). "만차 판정" 로직이 참조하는 용량값의 출처(수동 수집/2020년 스냅샷 등)는 **검증 필요** — 확보 전까지는 "보유대수" 위주로만 표기하고 "몇 자리 남았는지"는 노출하지 않는다.

### 2-3. 고장 신고/체크인 플로우

```
① 대상 특정          ② 유형 선택          ③ 사진(선택)        ④ 제출완료
QR 스캔 우선          잠김불가             +촬영/첨부           "접수됐습니다"
실패시 수동입력       배터리없음                                내 신고 보기 →
(정류장+번호)         외관파손
                     GPS 미인식
                     기타
```

- 사진은 **선택**이지만 첨부 시 신뢰도 가중치가 높아진다는 점을 화면에서 안내한다(현행 타슈 공식 신고는 사진 첨부가 필수로 무거운 절차인데, 여기서는 진입장벽을 낮추되 정확도는 사진으로 보강하는 방식으로 의도적으로 차별화한다).
- 유형 선택지는 1부 R4의 신고 분류 스키마(`fault_category` enum)와 1:1로 맞춘다 — 이용자가 고른 값이 그대로 Haiku 티어 분류 파이프라인 입력이 되어 별도 텍스트 파싱 없이 구조화된다.
- **중복신고 감지**: 동일 대상에 최근 신고가 이미 있으면 제출 전 "이미 접수된 신고가 있습니다 — 힘을 보태시겠어요?"로 안내해 중복 누적(1부 R2의 "독립 신고 N건 누적" 로직)을 사용자에게도 투명하게 보여준다.
- 상태: 입력중 / 제출중 / 성공 / 실패(재시도) / 중복감지

### 2-4. 정비사 체크인 화면

```
오늘의 점검(3)           →  탄방동 한사랑병원 도착      →  자전거 점검
1. 탄방동 한사랑병원(신뢰도 높음)    [GPS 체크인]              QR 스캔
2. 둔산동 샤크존(신뢰도 중간)                                 정상 / 경미결함(현장수리) / 중대결함(회수)
3. 시청역 8번출구                                            조치 메모 입력
                                                            [ 완료 ]
```

- 리스트는 AI 신뢰도 높은 순으로 정렬해 우선순위를 명시적으로 보여준다.
- 현장 도착은 GPS 체크인으로 확정(허위 체크인 방지), 이후 자전거별 점검 결과가 곧 1부 R2의 "정비사 체크인"이자 다른 두 체크인(이용자·AI)의 **검증 기준선**이 된다 — 정비사가 방금 정상 확인한 자전거에 대한 뒤늦은 사용자 오신고는 신뢰도를 낮게 매기는 근거로 쓰인다.
- 상태: 배정됨 / 이동중 / 현장도착 / 점검중 / 완료 / 보고서제출

### 2-5. 운영 대시보드

```
┌ 알림 피드 ──────┐┌ 선택 정류장 상세 ──────────────┐
│ 🔴 즉시(3)      ││ 탄방동 한사랑병원               │
│  탄방동...      ││ ─ 통합 타임라인 ─               │
│  둔산동...      ││ 13:10 사용자 신고 (잠김불가)     │
│ 🟡 관찰(12)     ││ 13:22 AI 신호 격상 (신뢰도 중간) │
│  ...            ││ [정비사 배정] [신고 반려]        │
│ ⚪ 로그(41)      │└─────────────────────────────┘
└────────────────┘┌ 오늘의 LLM 브리핑 ──────────────┐
                   │ "어제 이상 신호 18건 중 5건이    │
                   │  실제 신고와 일치했습니다..."     │
                   └─────────────────────────────┘
```

- 피드 등급은 1부 R3에서 제안한 임계값을 그대로 반영: `confidence ≥ 0.8` 즉시 알람, `0.5~0.8` 관찰목록, `<0.5` 로그만 적재.
- 상세 패널의 "통합 타임라인"이 이 화면의 핵심 — AI 신호·이용자 신고·정비 이력을 하나의 시간축에 겹쳐 보여줘, 운영자가 "이게 왜 이 판정을 받았는지"를 근거 데이터로 직접 추적할 수 있게 한다(2부 4장 AI 표기 윤리와 직결).
- **LLM 일일 브리핑**은 1부 R4에서 제안한 대로 Sonnet 티어가 하루치 이상 이력을 요약 생성한 결과이며, 브리핑 텍스트에도 동일하게 "숫자는 시스템값만 인용" 원칙이 적용된다.
- **오탐 피드백 검수 큐**: 알림별로 운영자가 정탐/오탐을 클릭해 라벨을 남기고, 이 라벨이 1부 R3의 Precision@k 평가와 임계값 재보정에 재사용된다.

## 3. 상태 시스템

### 3-1. 사용자 노출용 5단계 배지

과제에서 요구한 5개 배지를 기준으로 하되, 내부적으로는 1부 R2가 정의한 더 세밀한 상태머신(`normal → suspected → confirmed → resolved → normal`, 분기로 `dismissed`/`auto-cleared`)을 이 5개 배지로 압축해 노출한다.

| 배지 | 내부 상태 매핑 | 색상 원칙 | 아이콘 | 신뢰도 표기 |
|---|---|---|---|---|
| 이용가능 | normal | 녹색 | 체크 | 없음(확정 상태이므로) |
| 고장의심(AI) | ai_flagged / suspected(AI 단독) | 주황·노랑 계열 (절대 빨강 금지) | 물음표/경고 | 낮음·중간·높음 라벨 병기 필수 |
| 신고접수 | pending(이용자 신고, 임계치 미도달 또는 검증 대기) | 파랑 | 사람 아이콘 | AI 신뢰도와 별개로 "신고 N건" 숫자 표기 |
| 정비중 | assigned/arrived/inspecting | 보라·회색 | 렌치 | 없음(사람이 처리 중이므로 AI 추정 불필요) |
| 검증완료 | confirmed(고장 확정·조치완료) 또는 dismissed→normal 복귀 | 고장 확정은 빨강, 정상 확인은 다시 녹색 | 방패/확인마크 | 없음(사람이 최종 확인했으므로) |

### 3-2. 색·아이콘 원칙

1. **색상은 확정성(certainty)의 정도를 나타낸다.** 빨강(고장 확정)은 사람(정비사 또는 교차검증된 다중 신고)이 실제로 확인한 경우에만 쓴다. AI 단독 판정은 아무리 신뢰도가 높아도(1부 R4에서 `confidence: 높음`이 나오더라도) 주황·노랑 계열을 벗어나지 않는다 — 이는 1부 R2·R4에서 반복적으로 강조된 "절대 고장 확정으로 노출하지 않고 확인 필요 수준의 낮은 신뢰도 뱃지로만 표시" 원칙을 색상 규칙으로 그대로 옮긴 것이다.
2. **색만으로 구분하지 않는다.** 모든 배지는 형태가 다른 아이콘(체크/물음표/사람/렌치/방패)을 함께 쓴다 — 색맹 이용자를 위한 최소 요건이자 5장 접근성 원칙과 직결된다.
3. **신뢰도는 숫자(퍼센트)로 표시하지 않는다.** 1부 R4의 구조화 출력 스키마 자체가 `confidence`를 숫자가 아닌 `낮음/중간/높음` enum으로 설계한 이유는 JSON 스키마가 숫자 범위 제약을 지원하지 않기 때문만이 아니라, 이용자에게 거짓 정밀도(false precision)를 주지 않기 위함이기도 하다. UI도 이 설계를 그대로 따라 confidence를 항상 세 단계 라벨로만 노출한다.

## 4. AI 표기 윤리

1부 R4의 환각 억제 장치는 백엔드에서 끝나지 않는다. 프론트엔드에도 동일한 원칙이 이어져야 한다.

- **"AI 추정" 라벨 상시 노출**: AI가 산출한 모든 표시(보정된 대수, 고장의심 배지, 판정 문장)에는 예외 없이 "AI 추정" 태그가 붙는다. 원본 API 값과 AI 보정값은 항상 병기해, 어디까지가 실측이고 어디부터가 추정인지 이용자가 스스로 구분할 수 있게 한다.
- **근거는 항상 추적 가능해야 한다**: AI 판정 문장(`reasoning`)은 클릭하면 "무엇을 근거로 이렇게 판단했는지"(`referenced_fields`)를 확인할 수 있어야 한다. 화면에 노출되는 숫자는 예외 없이 시스템이 계산·검증한 값만 써야 하며, LLM이 생성한 자유 서술 텍스트 안에 새 숫자가 등장하면 프론트에서도 이를 신뢰하지 않고 시스템값과 대조해 불일치 시 해당 문장을 노출하지 않는다(1부 R4의 사후 정합성 검사를 UI 렌더링 단계에서도 한 번 더 건다는 뜻).
- **"보류"는 화면에서도 보류로 보인다**: `verdict = 보류`인 경우 "확인 중"으로 표기하고, 고장의심 배지나 자동 조치 트리거를 절대 발생시키지 않는다. 애매한 판정을 억지로 색깔 있는 배지로 밀어넣지 않는다.
- **오탐 시 사용자 피드백 루프**: 정류장 상세와 알림 카드 모두에 "이 판정이 정확한가요? [정확해요 / 아니에요]" 버튼을 상시 노출한다. "아니에요"를 선택하면 짧은 사유(실제로는 정상이었음 / 이미 수리됨 / 다른 문제였음)를 고르게 하고, 이 피드백은 1부 R3의 human-in-the-loop 라벨 축적 채널이자 대시보드의 오탐 검수 큐로 직결된다. 이 루프가 없으면 초기 몇 달간 모델을 재보정할 라벨 자체가 존재하지 않는다는 것이 1부 R3의 지적이므로, 이 버튼은 부가 기능이 아니라 시스템의 핵심 데이터 파이프라인 일부로 취급한다.

## 5. 접근성·모바일 우선 원칙

- **모바일 우선**: 이용자 앱은 이동 중·야외·한 손 조작을 기본 전제로 설계한다. 핵심 CTA(신고하기, QR 스캔)는 엄지 도달 영역에 배치하고 터치 타겟은 충분히 크게 잡는다.
- **색 단독 의존 금지**: 3장에서 정한 대로 모든 상태 배지는 색+아이콘+텍스트 라벨을 함께 쓴다.
- **명도 대비**: 라이트/다크 모드 모두에서 WCAG AA 이상의 대비를 유지한다.
- **스크린리더 대응**: 배지의 `confidence`, `verdict` 등은 시각 정보뿐 아니라 명시적 텍스트로도 읽혀야 한다(예: "AI 추정, 고장의심, 신뢰도 중간").
- **저시력·고령 이용자 고려**: 아이콘 단독 표기를 피하고 항상 짧은 텍스트 라벨을 병기하며, 시스템 폰트 크기 조절에 반응하도록 만든다.
- **현장 통신 취약 대응**: 정류장은 실내·지하·전파 음영 지역일 수 있다. 오프라인 상태에서는 마지막으로 캐시된 상태를 "동기화 안 됨" 표기와 함께 보여주고, 신고 제출은 로컬 큐에 쌓아 두었다가 재연결 시 자동 전송한다 — 이는 1부 R3의 `is_missing` 플래그(API 장애를 정류장 고장으로 오판하지 않기 위한 장치)를 클라이언트 쪽에서 대칭적으로 구현하는 것이기도 하다.

## 6. 디자인 토큰 개요

| 토큰 그룹 | 예시 | 비고 |
|---|---|---|
| `color.state.*` | `available`(녹색) / `suspected`(주황) / `reported`(파랑) / `servicing`(보라·회색) / `confirmed`(빨강) / `resolved`(녹색) | 3장 배지 체계와 1:1 매핑, 라이트/다크 모드 페어 정의 |
| `color.ai.badge` | 중립색 + 테두리 강조 | "AI 추정" 태그 전용 — 상태색과 별개 채널로 관리해 다른 배지와 혼동되지 않게 함 |
| `color.confidence.*` | `low`/`mid`/`high` (동일 색조의 채도 단계) | 숫자 대신 3단계로만 존재 |
| `type.badge` | 소형 semibold | 배지·라벨 전용 |
| `type.ai-reasoning` | 본문과 시각적으로 구분되는 스타일(배경색 박스 등) | "이 문장은 생성된 설명"임을 형태로도 드러냄 |
| `icon.state.*` | 체크/물음표/사람/렌치/방패 | 색맹 대응용 고정 아이콘 세트 |
| `space.touch-target` | 최소 터치 영역 스케일 | 모바일 우선 원칙 반영 |

이 토큰들은 세 화면(이용자 앱/정비사 모드/운영 대시보드) 전체에서 공유하는 단일 소스로 관리해, 예를 들어 "고장의심(AI)" 배지의 색이 지도 마커·정류장 상세·대시보드 알림 피드에서 제각각 달라지는 일이 없게 한다.

## 7. 디자인 산출물 목록과 제작 순서

배포 전략(1부 R2)상 초기에는 운영기관 제휴 없이 ①보정 앱 MVP 또는 ③공익 대시보드로 시작해, 실증을 쌓은 뒤 ②운영기관 제휴를 제안하는 단계적 접근이 유리하다고 판단했다. 디자인 산출물 순서도 이 로드맵을 그대로 따른다 — 즉 **이용자 체크인 + 시스템 자동탐지 조합을 먼저 검증**하고, 정비사 모드는 제휴 성사 이후로 미룬다.

1. **정보구조도(IA) & 사용자 플로우 다이어그램** — 본 문서 1장의 내용을 다이어그램으로 정식화 (완료 단계)
2. **로우파이 와이어프레임** — 우선순위: 지도 뷰 → 정류장 상세 → 신고 플로우(이 세 화면이 MVP의 전부) → 운영 대시보드 → 정비사 체크인(제휴 성사 후 착수)
3. **상태 시스템 & 배지 스펙 정의** — 3장의 컬러/아이콘 매트릭스를 컴포넌트 스펙 문서로 확정
4. **클릭 가능한 프로토타입(Figma)** — 신고 플로우를 최우선으로 제작(콜드스타트 해소가 1부 R2가 지적한 이용자 체크인 방식의 최대 리스크이므로, 진입장벽을 실제로 낮췄는지 프로토타입 단계에서 검증)
5. **디자인 시스템/토큰 라이브러리 구축** — 6장 토큰을 실제 컴포넌트 라이브러리(컬러/타이포/아이콘/버튼/배지/카드)로 구현
6. **하이파이 목업** — 전체 화면(운영 대시보드 포함) 정식화
7. **사용성 테스트** — 내부 QA → 소규모 실사용자 테스트. 정비사 모드는 제휴 파트너 확보 시점에 맞춰 별도로 진행
8. **개발 핸드오프** — 디자인 스펙 문서 + 컴포넌트 라이브러리 전달, `verdict`/`confidence`/`suspected_cause` 등 1부 R4의 JSON 스키마 필드명을 프론트 컴포넌트 프롭명과 그대로 일치시켜 번역 오차를 없앤다
