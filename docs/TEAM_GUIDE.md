# 타슈캐스트 — 역할별 실행 가이드 & API 계약서

> **문서 상태: 이전 사람팀 가이드.** 현재 API·enum·불변 규칙의 기준은 [`docs/v2/README.md`](./v2/README.md)와 `docs/v2/contracts/`다.

> 기획·디자인·개발이 **따로(병렬로) 일하기 위한 문서.** 각 트랙의 궁극적 목표·중요 포인트·산출물과, 트랙 사이를 잇는 계약(API·enum·문구)을 정의한다. 상세 배경은 `docs/PLAN.md`.
>
> **규칙 하나만 지키면 병렬 작업이 안 깨진다: 트랙 간 인터페이스(§4의 API 계약·enum)는 이 문서를 고쳐야만 바꿀 수 있다.** 임의로 필드를 추가·변경하지 않는다. (초안 단계에서 실제로 기획-디자인-개발이 서로 다른 상태 이름·없는 데이터를 전제하는 사고가 4건 발생했다 — `research/V1` 참조.)

## 0. 전체 구조 — 세 트랙이 만나는 지점

```
        [기획]
   기능 정의·완료기준·문구·enum 확정  ← 모든 것의 원본(Single Source)
      │                    │
      ▼                    ▼
   [디자인]  ◄──API 계약──►  [개발]
   화면 3개 + 배지 시스템        백엔드 파이프라인 + REST API
   (§4의 응답 JSON을 그대로       (§4의 스키마를 그대로 구현,
    화면에 바인딩, mock으로        mock 파일 먼저 커밋해서
    백엔드 완성 전에 개발)          프론트를 블로킹하지 않음)
      │                    │
      └────────┬───────────┘
               ▼
        E2E 통합 (1일차 밤) → 완료기준 검수 (2일차 오전, 기획 주관)
```

**동기화 시점 3회 (전원 모임은 이때만):**

| 시점 | 무엇을 확정하나 |
|---|---|
| ① 1일차 저녁 식사 전 | **계약 동결(freeze)** — §4의 엔드포인트·enum·문구 키를 확정. 이후 변경은 3인 합의로만 |
| ② 1일차 밤 | **E2E 통합** — mock을 실제 API로 교체, 데모 시나리오 1회 통주(通奏) |
| ③ 2일차 오전 | **완료기준 검수** — 기획이 §2.5(PLAN)의 수치 기준으로 테스트 10건 측정, 결과를 발표 자료에 기록 |

## 1. 기획 트랙

### 궁극적 목표
> **심사표 100점 구조를 통과하는 "스토리"와, 측정 가능한 "완료기준"의 소유자가 된다.**
> 데모가 아무리 멋져도 서류·발표가 심사 항목과 어긋나면 진다. 반대로 기능이 소박해도 문제 정의(30점)와 완료기준(12점)이 수치로 서 있으면 이긴다.

### 중요 포인트
1. **문제 문장은 템플릿에 끼워서 완성한다** (PLAN §2.1): `[핵심 사용자]는 [대전의 구체적 장소/시간]에서 [불편]을 겪는다...` — "대전 시민"이 아니라 **"충남대 통학생, 궁동~정문 구간, 평일 오전"**까지 좁힌다.
2. **오늘 중 현장 인터뷰 3명** — 캠프장이 충남대라서 정류장이 도보권이다. "앱 믿고 갔다가 허탕친 적 있나요?" 한 질문이면 된다. 인터뷰 인용문은 문제정의 30점의 실탄이다.
3. **완료기준은 전부 숫자로**: 테스트 10건 중 8건 성공, 10초 이내 응답, 3번 이하 조작, E2E 30초. 검수(동기화 ③) 때 이 수치로 직접 측정한다.
4. **비목표를 지키는 경비원 역할**: 로그인·QR·사진첨부·운영자 대시보드가 스코프에 기어들어오면 자르는 게 기획의 일이다. Core 1 + Supporting 2에서 끝.
5. **AI 필연성 서사**: "AI를 빼면 서비스가 성립하지 않는 3지점"(예측·감지·해석)을 발표에 명시 — 본선 AI활용도 25점 직결.
6. **제출 문서 정정**: 기존 타슈캐스트 아이디어 문서의 "반납 성공 가능성"과 "API가 거치대 수 제공" 문구는 실측과 어긋난다(거치대 용량 필드 없음 [확인]) — 제출서 갱신 시 수정.

### 산출물 (마감)
- 문제 문장 + 인터뷰 3건 메모 (1일차 저녁)
- **UI 문구 사전**: 배지 이름·고지 문구·버튼 라벨 확정본 (1일차 저녁, §4.3의 enum과 1:1)
- 발표 덱 + 90초 데모 시나리오 대본 (2일차 오후)
- 완료기준 측정 기록표 (2일차 오전)

## 2. 디자인 트랙

### 궁극적 목표
> **화면 3개만으로 "추정(AI)과 확인(사람)의 분리"가 눈에 보이게 만든다.**
> 이 서비스의 차별점은 예측 그 자체가 아니라 *정직한 표기*다 — AI가 어디까지 아는지, 어디부터 모르는지가 화면에서 구분되면 심사위원에게 "생각한 팀"으로 보인다.

### 중요 포인트
1. **새 숫자를 만들지 않는다** (철칙): "보정된 실제 이용가능 N대" 같은 계산값 표기 금지. 화면에는 ① API 원본 수량 ② 신뢰도 배지(낮음/중간/높음) ③ 설명 문장만. (초안에서 이 위반이 최다 심각도 지적이었다 — V1 치명1.)
2. **빨강은 사람이 확인했을 때만**: AI 단독 판정은 신뢰도가 '높음'이어도 주황/노랑을 벗어나지 않는다. 색+아이콘+텍스트 3중 표기(색맹 대응).
3. **신뢰도는 퍼센트 금지, 3단계 라벨만** — 거짓 정밀도를 주지 않는다.
4. **모든 AI 산출물에 "AI 추정" 태그** + 화면 하단 상시 고지 "타슈 공식 서비스 아님 · 공공데이터 기반 추정 정보".
5. **상태별 화면을 빠뜨리지 않는다**: 로딩 / 정보지연(`is_missing`) / 분석중(콜드스타트 — AI 배지 숨김) / 보류("확인 중") / 신고 중복 감지. 데모에서 이 상태들이 자연스럽게 지나가면 완성도 점수가 올라간다.
6. **작업 우선순위 = 데모 동선 순서**: 지도 뷰 → 정류장 상세 → 신고 플로우. 하이파이보다 세 화면의 상태 완결성이 먼저다.

### 산출물 (마감)
- 화면 3개 와이어프레임 + 상태 변형 (1일차 저녁)
- 배지·색·아이콘 스펙 1장 (1일차 저녁 — §4.3 enum과 1:1 매핑 명시)
- Figma 클릭 프로토타입 or 프론트 구현 지원 (1일차 밤~2일차)
- 발표 덱 비주얼 (2일차 오후)

## 3. 개발 트랙

### 궁극적 목표
> **시연장 네트워크가 죽어도 90초 데모가 도는 E2E 파이프라인.**
> 화려한 모델보다 "데이터 수집 → 등급 산출 → Claude 설명 → 신고 → 배지 변경"이 끊기지 않고 한 바퀴 도는 것이 전부다. `--demo-mode`는 보험이 아니라 1급 기능이다.

### 중요 포인트
1. **데이터 확보가 첫 작업** (순서 고정): ① data.go.kr 15109253 활용신청(자동승인) → ② 대여이력 CSV 4,168,667행 다운로드(`cp949`!) → ③ 타슈 직접 API는 신청만 하고 잊는다(수동승인) → ④ 스냅샷 JSON 저장해 demo-mode 데이터 확보.
2. **`x_pos`가 위도, `y_pos`가 경도다** — 이름과 반대. 백엔드에서 `lat`/`lng`로 변환해 내보내고, 프론트는 절대 원본 필드명을 보지 않게 한다(§4.2 계약에 반영됨).
3. **API 장애 ≠ 고장**: 폴링 실패는 `is_missing=true`로 구분 저장. 이 플래그가 없으면 수집 장애가 "고장 의심"으로 둔갑한다.
4. **모델은 기준선부터**: 시간대·요일 평균 전이율로 등급을 먼저 완성 → 홀드아웃(이력 마지막 1주)으로 기준선 대비 개선 수치 기록(발표용) → 여유 있을 때만 LightGBM.
5. **LLM 역할 제한**: 수치는 전부 백엔드가 계산해 주입, Claude는 해석·분류만. `output_config.format`으로 스키마 강제, `referenced_fields` 사후 검증, 실패 시 템플릿 문구 폴백. 판정 = `claude-sonnet-5`, 신고 분류 = `claude-haiku-4-5`.
6. **mock 파일을 먼저 커밋한다**: §4의 응답 예시 JSON을 그대로 `mock/` 디렉터리에 넣어 프론트/디자인이 백엔드 완성 전에 화면을 붙일 수 있게 한다. **mock과 실제 응답의 스키마가 달라지는 순간 통합(동기화 ②)이 깨진다.**

### 산출물 (마감)
- `hourly_agg` 집계 + `stations.json` 메타 (1일차 오후)
- mock 파일 4종 커밋 (1일차 오후 — 프론트 블로킹 해제 시점)
- `GET /forecast` 동작 + Claude 연동 (1일차 저녁~밤)
- 신고 API + 상태머신 (2일차 오전)
- demo-mode 스위치 + 리허설 체크리스트 통과 (2일차 오후)

## 4. API 계약 (트랙을 잇는 인터페이스)

### 4.1 화면 ↔ 엔드포인트 매핑

| 화면 (디자인) | 엔드포인트 (개발) | 갱신 방식 | mock 파일 |
|---|---|---|---|
| 화면1 지도 뷰 | `GET /api/v1/stations` | 진입 시 + 30초 폴링 or 수동 새로고침 | `mock/stations.json` |
| 화면2 정류장 상세 | `GET /api/v1/stations/{id}/forecast?eta_min=15` | 진입 시 1회 | `mock/forecast_ST0031.json` |
| 화면2 피드백 버튼 | `POST /api/v1/anomaly-events/{id}/user-feedback` | 클릭 시 | `mock/feedback_ok.json` |
| 화면3 신고 플로우 | `POST /api/v1/reports` → `GET /api/v1/reports/{id}` | 제출 시 / 내 신고 조회 | `mock/report_created.json` |

### 4.2 응답 스키마 (mock = 실제 응답, 바이트 단위로 동일 구조)

**`GET /api/v1/stations`**

```json
{
  "as_of": "2026-07-21T14:05:00+09:00",
  "data_source": "live",
  "stations": [
    {
      "station_id": "ST0031",
      "name": "충남대 정문",
      "lat": 36.36215,
      "lng": 127.34462,
      "parking_count": 2,
      "is_missing": false,
      "badge": "ai_suspected",
      "confidence": "중간",
      "forecast_grade": "보통",
      "report_count": 1
    }
  ]
}
```
- `data_source`: `"live" | "demo"` — demo-mode면 화면 구석에 "샘플 데이터" 워터마크(디자인 처리).
- `lat`/`lng`는 백엔드가 이미 변환한 값. 프론트에 `x_pos`/`y_pos`는 절대 노출하지 않는다.
- `is_missing=true`면 `badge`/`confidence`/`forecast_grade`는 `null` → 회색조 "정보 지연" 표기.

**`GET /api/v1/stations/{id}/forecast?eta_min=15`**

```json
{
  "station_id": "ST0031",
  "as_of": "2026-07-21T14:05:00+09:00",
  "eta_min": 15,
  "parking_count": 2,
  "forecast": { "grade": "보통", "baseline": "요일·시간대 평균 전이율" },
  "ai_verdict": {
    "verdict": "고장_의심",
    "confidence": "중간",
    "reasoning": "이 정류장은 지난 38분간 대여·반납 변동이 없었고, 인근 정류장은 정상 순환 중입니다. 거치대·자전거 이상이 의심됩니다.",
    "recommended_action": "도보 4분 거리 궁동 로데오거리 이용을 권합니다.",
    "referenced_fields": ["stagnation_minutes", "neighbor_turnover"],
    "llm_generated": true
  },
  "signals": { "stagnation_minutes": 38, "neighbor_turnover_ok": true },
  "alternative": { "station_id": "ST0018", "name": "궁동 로데오거리", "walk_extra_min": 4, "grade": "높음" },
  "recent_reports": [ { "category": "잠금불가", "at": "2026-07-21T13:10:00+09:00" } ],
  "anomaly_event_id": "ae_01H..."
}
```
- `ai_verdict`가 `null`이면 → 콜드스타트: "분석 중 · 데이터 축적 필요" 라벨.
- `verdict="보류"`면 → 배지 없이 "확인 중" 표기 (색깔 배지 금지).
- `llm_generated=false`면 폴백 템플릿 문구라는 뜻 — 화면 표기는 동일, "AI 추정" 태그도 동일 유지.
- `alternative`는 `null` 가능(대안 없음) — 그 경우 대안 카드 자체를 숨김.
- `anomaly_event_id`는 피드백 버튼(`user-feedback`)이 참조할 키. AI 신호가 없으면 `null` → 피드백 버튼 숨김.

**`POST /api/v1/reports`** — 요청:

```json
{ "station_id": "ST0031", "category": "잠금불가", "text": "QR 찍어도 잠금이 안 풀려요", "device_id": "dvc_a1b2c3" }
```
응답 `201`:
```json
{
  "report_id": "rp_01H...",
  "status": "접수",
  "classified": { "category": "잠금불가", "urgency": "보통" },
  "station_badge_now": "reported",
  "duplicate_hint": false
}
```
- `duplicate_hint=true`면 "이미 접수된 신고가 있어요 — 힘을 보태시겠어요?" 플로우로 분기.
- `classified`는 Haiku 분류 결과. text 없으면 사용자가 고른 category 그대로 에코.

**`GET /api/v1/reports/{id}`**

```json
{
  "report_id": "rp_01H...",
  "station_id": "ST0031",
  "status": "의심",
  "timeline": [
    { "at": "2026-07-21T13:10:00+09:00", "status": "접수" },
    { "at": "2026-07-21T13:22:00+09:00", "status": "의심", "note": "AI 신호와 교차 확인됨" }
  ]
}
```

**`POST /api/v1/anomaly-events/{id}/user-feedback`** — 요청:
```json
{ "device_id": "dvc_a1b2c3", "accurate": false, "reason": "실제로는 정상이었어요" }
```
응답 `200`: `{ "recorded": true }` (기기당 이벤트별 1회, 초과 시 `409`).

### 4.3 enum ↔ UI 매핑 (기획이 소유, 세 트랙 공용)

| enum (개발 코드값) | UI 라벨 (기획 문구) | 색/아이콘 (디자인) |
|---|---|---|
| `badge: normal` | 이용가능 | 녹색 · ✓ |
| `badge: ai_suspected` | 고장의심(AI 추정) | **주황** · ? + 신뢰도 라벨 필수 |
| `badge: reported` | 신고접수 | 파랑 · 👤 + "신고 N건" |
| `badge: confirmed` | 고장확인 | 빨강 · 🛡 (사람 검증 후에만) |
| `badge: analyzing` | 분석 중 | 회색 · … (콜드스타트) |
| `confidence: 낮음/중간/높음` | 그대로 | 동일 색조 채도 3단계, 퍼센트 금지 |
| `forecast.grade: 낮음/보통/높음` | "15분 뒤 대여 가능성 {값}" | 텍스트 위주 |
| `verdict: 고장_의심/정상_변동/보류` | 의심/— /"확인 중" | 보류는 배지 없음 |
| `report category: 잠금불가/배터리없음/외관파손/수량불일치/기타` | 신고 화면 선택지 그대로 | 라디오 5개 |
| `report status: 접수/의심/확인/해소/기각` | 내 신고 타임라인 | 파랑→주황→빨강→녹색→회색 |

### 4.4 내부 계약 (개발 트랙 내부, 참고)

- **백엔드 ↔ 공공데이터**: data.go.kr 타슈정보(5분 폴링, 실패 시 `is_missing`) + 대여이력 CSV 1회 집계. 원본 필드(`x_pos`/`y_pos` 등)는 수집 계층에서만 존재.
- **백엔드 ↔ Claude**: 판정 `claude-sonnet-5` / 분류 `claude-haiku-4-5`, `output_config.format` 스키마 강제(PLAN §4.4), 시스템 프롬프트 `cache_control` 캐싱, 실패 시 폴백 문구. **LLM 응답은 §4.2의 `ai_verdict` 형태로 검증을 통과한 뒤에만 프론트로 나간다** — 검증 실패 건은 `llm_generated:false` 폴백으로 대체.
