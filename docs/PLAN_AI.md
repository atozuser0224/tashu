# 타슈캐스트 V2 — AI 수행 전제 기획 (기한 무제약)

> **문서 상태: 탐색적 이전안.** 현재 제품·데이터·아키텍처 기준은 [`docs/v2/README.md`](./v2/README.md)다. 본 문서의 과잉 구현 항목은 검증 게이트를 통과한 경우에만 재검토한다.

> **전제 변경**: `PLAN.md`(해커톤판)는 "사람 3트랙 + 2일"이 제약이었다. 이 문서는 **기획·디자인·개발 세 트랙을 전부 AI 에이전트가 수행하고, 기한이 없다**고 가정하고 처음부터 다시 짠 기획이다.
> 변하지 않는 것: 타슈 API의 데이터 제약([확인] 사실들 — `PLAN.md` §0), 오탐→신뢰 손상 문제, 비공식 서비스라는 법적 위치. AI가 아무리 일해도 이 제약들은 노동력으로 뚫리지 않는다.
>
> 참고: **이 리포의 계획서 자체가 이 모델의 실증이다** — 리서치 4건·초안 3건·적대적 검증 1건을 Sonnet 5 에이전트 8개가 수행했고, 검증 에이전트가 사람이 놓칠 트랙 간 모순 10건을 잡아냈다(`research/V1`). 아래 조직 설계는 그 패턴의 일반화다.

## 0. 전제가 바뀌면 무엇이 바뀌나

| | 해커톤판 (PLAN.md) | AI 수행판 (이 문서) |
|---|---|---|
| 희소 자원 | 사람의 시간 (2일 × 3~5명) | **실세계 피드백** — 고장 확정 라벨, 이용자 채택, 기관 제휴 |
| 설계 원리 | 스코프 축소 (Core 1 + Supporting 2) | **검증 게이트** (만드는 건 싸다, 틀린 걸 배포하는 게 비싸다) |
| 순서 제약 | "경량 버전 먼저, 여유 되면 고도화" | 순서 제약 소멸 — 병렬 동시 구축. 단 **달력 시간이 필요한 것만 남는다** (§4) |
| 트랙 간 계약 | 문서 + 회의 3번 | **기계가 읽는 계약 파일** (openapi.yaml, enums, tokens) + 적대적 검증 루프 |
| 실패 모드 | 미완성 | **과잉 구현·환각·드리프트** — 기한이 없을수록 위험 (§6) |

**핵심 통찰**: AI 노동력이 무제한이면 "무엇을 만들 수 있나"는 더 이상 기획 질문이 아니다. 기획 질문은 세 가지로 좁혀진다 —
1. **데이터가 원리적으로 허락하는 것은 어디까지인가** (정류장 집계값의 해상도 한계)
2. **사람의 확인이 반드시 필요한 지점은 어디인가** (고장 확정, 가치 판단, 대면 협상)
3. **달력 시간이 필요한 것은 무엇인가** (데이터 축적, 라벨 축적, 신뢰 축적)

## 1. 목표 재설정 — 기한이 없으므로 "일정"이 아니라 "상태"로 정의

**North Star (최종 상태)**: 대전 타슈 전 정류장(1,000~1,500개소)에 대해 실질 이용가능성 정보가 공식 앱보다 신뢰할 수 있게 제공되고, 모든 고장이 `신고 → 확인 → 해소`로 추적되며, 운영기관이 이 데이터를 정비 우선순위에 실제로 사용하고, 구조가 타 지자체(동일한 "집계값만 공개" 패턴 [확인 R2])로 이식 가능한 상태.

성공의 단위는 마일스톤 날짜가 아니라 **상태 전이 조건**이다(§4). 어떤 상태도 "○주 안에"가 아니라 "○○이 달성되면"으로만 넘어간다.

## 2. AI 조직 설계 — 세 트랙을 AI가 어떻게 수행하나

각 트랙의 **궁극적 목표는 사람판과 동일**하다(목표는 수행 주체와 무관). 바뀌는 것은 수행 방식과, 사람이 남는 지점이다.

### 2.1 AI 기획팀

> 궁극 목표(불변): **가설-검증 루프의 소유자** — 무엇이 문제이고, 무엇이 되면 성공인지의 정의를 데이터로 유지한다.

| 에이전트 | 역할 | 트리거 |
|---|---|---|
| 리서치 에이전트 | 공공데이터 갱신·타슈 공지·민원/뉴스·타 지자체 사례 상시 모니터링 → 사실 대장(fact ledger) 갱신. [확인]/[추정] 태그 체계 유지 | 일 1회 + 데이터셋 갱신 이벤트 |
| 지표 에이전트 | KPI(Precision@k, 리드타임, 오탐률, 캘리브레이션) 해석 → 주간 보고 + 임계값 조정 제안 | 주 1회 + 지표 급변 알람 |
| 실험 설계 에이전트 | A/B·섀도 모드 실험 설계, 표본 크기·중단 조건 산정 | 백로그에 가설이 쌓일 때 |
| 문서 에이전트 | B2G 제안서·공익 대시보드 리포트·보도자료 초안 생성 (트랙레코드 데이터 자동 인용) | 상태 게이트 접근 시 |

- **AI가 대체하는 것**: 리서치 노동, 지표 해석, 문서 작성, 실험 설계. 사람 기획자의 "감"을 상시 데이터 모니터링으로 대체.
- **사람이 남는 지점**: ① 가치 판단(예: 오탐률 5%에서 알람을 더 낼 것인가 덜 낼 것인가는 트레이드오프 선택이지 계산이 아님) ② 대전교통공사 대면 협상 ③ 비목표 변경 승인(스코프 확장은 반드시 사람 결재).
- **산출 루프**: 사실 대장 갱신 → 백로그 이슈 발행(구조화 포맷) → 개발팀 에이전트가 소비. 모든 이슈는 "완료기준(측정 가능)" 필드가 채워져야 발행됨 — 기획 에이전트의 시스템 프롬프트에 강제.

### 2.2 AI 디자인팀

> 궁극 목표(불변): **"추정(AI)과 확인(사람)의 분리"가 모든 화면·모든 상태에서 지켜지는 UI.**

- **AI가 대체하는 것**:
  - **와이어프레임→Figma→코드 파이프라인을 코드 직행으로 단축.** AI는 코드로 디자인한다 — 디자인 시스템의 실체는 React 컴포넌트 라이브러리 + `tokens.json`이며, 목업은 스토리북 스크린샷으로 대체.
  - **상태 조합 전수 생성**: 사람 디자이너는 5~6개 대표 상태를 그리다 빠뜨리지만(해커톤판 V1 권고9 — 콜드스타트 화면 누락이 실제 사례), AI는 `화면 × 상태(로딩/정보지연/분석중/보류/정상/의심/신고/확인) × 라이트·다크 × 접근성` 매트릭스를 전수 생성하고 스크린샷 리그레션 테스트로 고정한다.
  - 접근성(WCAG AA 대비, 스크린리더 라벨) 자동 검사·자동 수정.
- **사람이 남는 지점**: ① **실사용자 사용성 테스트** — 사람의 반응은 사람에게서만 나온다. 충남대 이용자 5명 테스트 같은 활동은 AI가 설계·기록하되 수행은 사람이 참여 ② 미적 방향 선택 — AI가 4개 시안(팔레트·타이포·레이아웃)을 제안하고 사람이 픽(생성 다양성은 프롬프트로 강제).
- **불변 원칙 (시스템 프롬프트 상주)**: 새 숫자 표기 금지 · AI 단독 판정에 빨강 금지 · 신뢰도 퍼센트 금지 · "AI 추정" 태그 상시 · `verdict=보류`는 배지 없음. — 이 원칙들은 디자인 에이전트가 위반할 수 없는 제약으로 컴포넌트 레벨에 하드코딩한다(예: `<ConfidenceBadge>`는 애초에 숫자 prop을 받지 않음).

### 2.3 AI 개발팀

> 궁극 목표(불변): **신뢰 가능한 자율 파이프라인** — "코드 생성"이 아니라 "검증 루프 통과"가 완료의 정의.

| 에이전트 | 역할 |
|---|---|
| 구현 에이전트 (N개 병렬) | 이슈 단위 구현 — 수집기, 피처 엔진, 하이브리드 스코어러(축A LightGBM 잔차 + 축B 정체 규칙), LLM 서비스, API 서버, 프론트 |
| 리뷰 에이전트 | **적대적 검증** — 구현이 계약(openapi.yaml, enums)과 불변 원칙을 위반하는지 반박 시도. V1 패턴의 상시화 |
| 테스트 에이전트 | 유닛·E2E·홀드아웃 평가 리그레션. 데모 시나리오의 자동 재생 |
| 모델 에이전트 | 재학습 주기 실행, 드리프트(PSI/ADWIN) 감시, 섀도 평가 → 승격 제안 |
| 운영 에이전트 | 폴러 헬스·`is_missing` 비율·LLM 캐시 히트율·비용을 감시, 이상 시 원인 해석 리포트 생성 |

- **사람이 남는 지점**: ① 배포 승인 게이트(운영 환경 반영은 사람 클릭) ② 시크릿·API 키 관리 ③ 비용 상한 설정(에이전트는 상한 내에서만 자율).
- **기한 무제약의 실질 효과**: 해커톤판의 "경량 먼저" 순서가 사라진다. 수집기·하이브리드 탐지·LLM 레이어·이용자 앱·운영 대시보드를 **병렬 동시 구축**한다. 단 §4의 달력 시간 항목은 병렬화 불가.

## 3. 기능 스코프 — 해커톤 축소분 복원 + AI 노동력 전제 신규

### 3.1 복원 (해커톤에서 잘랐던 것)

| 기능 | 해커톤판 처리 | V2 |
|---|---|---|
| 전 정류장 커버리지 | 충남대 주변 5~10곳 | 1,000~1,500개소 전체 (S0 데이터 축적 후) |
| 하이브리드 이상탐지 | 축B 규칙만 | 축A(LightGBM pooled 잔차) + 축B + STL 백업 + river 온라인 보조 (R3 설계 전문) |
| QR 스캔·사진 첨부 신고 | Stretch로 강등 | 정식 기능 |
| 운영자 대시보드·일일 브리핑 | Stretch | 정식 (Sonnet 판정 + Batch API 브리핑, R4 설계) |
| 정비사 모드 | 범위 외 | 설계·구현 완료 상태로 대기, **활성화는 제휴 게이트(S3)에 종속** — 노동력이 아니라 권한이 병목 |
| 날씨·외생변수 결합 | Stretch | 정식 피처 |

### 3.2 신규 (AI 노동력이라서 가능해지는 것)

1. **사진 신고 자동 판독**: 신고 첨부 사진을 Claude 비전으로 판독 — 파손 유형 분류(체인 이탈/안장 파손/거치대 문제 등), 사진-신고유형 불일치 감지(오신고 필터). 사람 검수 없이 1차 분류가 즉시 반영되고, 저신뢰 건만 검수 큐로.
2. **대화형 신고 접수**: 폼 대신 "자전거가 안 빌려져요" 자유 발화 → 에이전트가 정류장·유형·상황을 질문으로 구조화. 폼 이탈률을 낮추는 접수 채널.
3. **정류장별 주간 리포트 자동 발행**: 공익 대시보드(배포형태 ③)의 콘텐츠를 문서 에이전트가 상시 생성 — "이번 주 유성구에서 가장 오래 방치된 이상 신호" 같은 공공 데이터 스토리.
4. **B2G 제안서 자동 갱신**: 트랙레코드(Precision, 리드타임, 신고 해소 건수)가 쌓일 때마다 제안서의 실증 섹션이 자동 업데이트 — 제휴 협상(사람)이 언제든 최신 자료로 들어감.
5. **셀프 튜닝 임계값**: 오탐 피드백이 쌓이면 정류장·시간대별 임계값을 실험 설계 에이전트가 제안 → 섀도 검증 → 사람 승인 → 반영되는 반자동 루프.

### 3.3 여전히 하지 않는 것 (비목표 — AI 노동력과 무관한 제약)

- 공식 타슈 앱·서버 수정 (권한 없음), 개별 자전거 GPS 우회 수집 (약관 위반 소지), 결제·요금 분쟁 관여, "고장 확정"의 AI 단독 판정 (원리적으로 불가 [확인] — 사람 검증 없이는 `confirmed` 승격 금지 유지).
- **스코프 확장은 사람 결재 없이 금지** — 기한이 없는 AI 팀의 1순위 실패 모드가 스코프 폭발이다(§6).

## 4. 상태 게이트 로드맵 — 날짜 없음, 달성 조건만

기한을 없애도 **달력 시간이 필요한 것**은 남는다: ① 시계열 데이터 축적 ② 고장 확정 라벨 축적 ③ 이용자·기관의 신뢰 축적. 이 세 가지만이 진짜 일정이며, 나머지는 전부 병렬 구축 대상이다.

| 상태 | 진입 조건 (이전 상태 달성) | AI 팀 활동 | 사람 게이트 | 전이 조건 (exit) |
|---|---|---|---|---|
| **S0 수집** | 즉시 시작 | 수집기·DB·전 기능 병렬 구축, 대여이력 CSV 학습, 섀도 탐지 가동 | API 키 발급, 배포 승인 | 전 정류장 스냅샷 축적 + `is_missing` 비율 기준 이내 연속 2주 [달력 시간] |
| **S1 감지 가동** | S0 | 하이브리드 스코어 실가동, 이용자 앱·신고 채널 공개, LLM 판정·브리핑 가동 | 공개 승인, 오탐 정책(알람 예산) 확정 | 신고-알람 매칭 표본 확보, Precision@k 최초 측정 성립 [라벨 축적 = 달력 시간] |
| **S2 시민 채택** | S1 | 오탐 피드백 루프·셀프 튜닝, 대화형 신고, 주간 리포트 발행, 사용성 테스트 설계 | 사용성 테스트 참여, 트레이드오프 판단 | 재방문 이용자·신고 건수가 자생적 증가 추세 + 캘리브레이션(신뢰도 등급 vs 실제 확정 비율) 검증 통과 |
| **S3 기관 채택** | S2 | B2G 제안서 자동 갱신, 정비사 모드 활성화 준비(배정 API·GPS 체크인·`assigned/arrived/inspecting` 상태 — V1 치명3 반영분), 데이터 품질 SLA 리포트 | **제휴 협상(전적으로 사람)**, MOU 체결 | 정비 체크인 데이터 유입 개시 → `confirmed`의 ground truth 확보 |
| **S4 이식** | S3 | 타 지자체 어댑터(수집기 추상화 — GBFS·따릉이 API 등), 지역별 모델 재학습 자동화 | 대상 지역 선정, 현지 파트너 | 두 번째 도시에서 S1 재현 |

주: S3(제휴)는 S2와 병렬 시도 가능하나, 트랙레코드 없이는 성사 확률이 낮다는 것이 리서치 판단(R2 — 실증 후 제안하는 단계적 접근, 대구-카카오T바이크 선례).

## 5. 트랙 간 인터페이스 — 회의를 계약 파일로 대체

사람판(TEAM_GUIDE.md)의 "전원 모임 3회"는 AI판에서 **기계가 읽는 계약 + 상시 검증**으로 대체된다.

### 5.1 Single Source 계약 파일 (리포가 곧 회의실)

| 파일 | 내용 | 소유 | 소비 |
|---|---|---|---|
| `contracts/openapi.yaml` | REST 전체 스펙 — TEAM_GUIDE §4.2의 5개 + D3 확장분(정비사 배정 `POST /anomaly-events/{id}/assign`, `GET /mechanic/assignments`, 대시보드 `GET /anomaly-events`, 운영자 피드백, 브리핑 `GET /briefings/daily`) | 기획 에이전트 (변경 제안) | 개발·디자인 에이전트 (코드 생성 원본) |
| `contracts/enums.json` | badge/verdict/confidence/category/report status ↔ UI 라벨 ↔ 색 토큰 | 기획 | 전 트랙 |
| `contracts/tokens.json` | 디자인 토큰 (색·타이포·간격·상태색) | 디자인 | 프론트 |
| `contracts/llm_schemas/` | Claude 구조화 출력 스키마(판정·분류·브리핑) + 프롬프트 버전 | 개발(모델 에이전트) | LLM 서비스 |
| `contracts/invariants.md` | 불변 원칙 목록 (새 숫자 금지, 빨강 규칙, 보류 강제, confirmed 사람검증 필수...) | 기획 | **리뷰 에이전트의 검증 체크리스트** |

- 계약 변경 = PR. 리뷰 에이전트가 하위호환("nullable + 기본 숨김"만 허용)을 기계 검증하고, 위반 시 자동 반려. 프론트·백엔드·목업은 같은 PR에서 계약으로부터 재생성.
- mock은 별도 관리하지 않는다 — openapi.yaml에서 자동 생성 (사람판의 "mock 드리프트" 리스크 원천 제거).

### 5.2 에이전트 간 핸드오프 규약

```
[기획] 사실 대장/지표 → 이슈 발행 (완료기준 필수 필드)
    → [개발] 구현 에이전트가 이슈 단위 브랜치·PR
        → [리뷰] 적대적 검증: 계약 위반·invariants 위반·리서치에 없는 사실 발명 여부를 반박 시도
            → 통과 시에만 [사람] 배포 승인 게이트
    → [디자인] 계약 변경 시 컴포넌트·스크린샷 테스트 자동 재생성 → 동일 리뷰 루프
[운영] 지표 이상 → 원인 해석 리포트 → 기획 에이전트의 사실 대장으로 환류 (루프 폐쇄)
```

**적대적 검증이 접착제다.** 사람판에서 트랙 간 모순은 회의에서 잡았지만, AI판에서는 산출물마다 리뷰 에이전트가 "이 구현/화면이 계약·불변 원칙·리서치 사실과 모순되는가"를 상시 검증한다. 이 세션의 V1(치명 4·권고 6 적발)이 이 루프의 실증이다.

### 5.3 LLM 운영 구성 (서비스 런타임)

| 용도 | 모델 | 비고 |
|---|---|---|
| 신고 분류·사진 1차 판독·트리아지 | `claude-haiku-4-5` | 실시간, enum 강제 |
| 이상 판정·근거 생성·대화형 접수 | `claude-sonnet-5` | 실시간, 프롬프트 캐싱 |
| 저신뢰 반복 건·주간 리포트·B2G 문서 | `claude-opus-4-8` | 온디맨드/Batch(50% 할인) |

서비스 LLM 비용은 이상 후보만 호출하는 구조로 전 정류장 기준 월 $240~720 [추정, R4 재환산] — 개발·기획·디자인 **에이전트 운영 비용은 이와 별도**이며 사용량 기반이라 상한(사람 게이트)으로 통제한다.

## 6. 리스크 재평가 — AI 수행 특유의 실패 모드

| 리스크 | 실증 사례 | 대응 |
|---|---|---|
| **환각 (없는 데이터 발명)** | V1 치명1 — 디자인 초안이 존재하지 않는 "고장 대수" 산출값을 전제로 화면을 그림 | 리뷰 에이전트가 "리서치 사실 대장에 없는 필드·수치" 검출을 전용 체크로 수행. LLM 런타임에는 `referenced_fields` 사후 대조 |
| **트랙 간 드리프트** | V1 치명2·5 — 초안 3부가 서로 다른 마일스톤 정의·상태 이름 사용 | 계약 파일 Single Source + 계약 위반 자동 반려. 문서보다 기계 검증 |
| **과잉 구현 (스코프 폭발)** | 기한 없는 AI 팀의 구조적 경향 | 비목표 목록·invariants를 모든 에이전트 시스템 프롬프트에 상주. 신규 스코프는 사람 결재 필수. "이슈에 완료기준 없으면 착수 금지" |
| **검증 없는 자신감** | 진행 보고가 실제 결과와 다를 위험 | 완료 주장 = 테스트 에이전트의 통과 로그 링크 필수. 지표는 운영 에이전트가 독립 산출 |
| **비용 폭주** | 에이전트 무한 루프·과잉 재시도 | 트랙별 토큰 예산 상한, 초과 시 정지 + 사람 알림 |
| **사람 게이트 병목화** | 모든 것이 사람 승인에 몰려 자율성 상실 | 게이트를 4개로 한정: 배포 승인 / 스코프 변경 / 트레이드오프 판단 / 대외 발신. 그 외는 사후 감사로 |

데이터·법률 리스크(호출 한도, 오탐 신뢰 손상, 비공식 서비스 고지)는 해커톤판 §2.7과 동일하게 유지 — 수행 주체가 바뀌어도 사라지지 않는다.

## 7. 사람에게 남는 역할 (최소 집합)

| 역할 | 왜 AI로 대체 불가 | 개입 빈도 |
|---|---|---|
| 가치·트레이드오프 판단 | "알람을 더 낼 것인가"는 계산이 아니라 선택 | 주 1회 수준 |
| 배포·스코프·대외발신 승인 | 책임의 귀속 | 이벤트성 |
| 대전교통공사 협상 (S3 게이트) | 신뢰는 대면에서 형성 | 협상 기간 집중 |
| 실사용자 테스트 참여·현장 검증 | 사람의 반응·현장의 실물은 사람만 확인 | 상태 전이 시 |
| API 키·시크릿·비용 상한 | 보안·재무 통제 | 설정 시 |

이 표가 곧 이 기획의 결론이다: **AI가 셋 다 일하면, 사람의 일은 "만드는 것"에서 "결정하고 검증하고 신뢰를 트는 것"으로 압축된다.** 그리고 로드맵의 속도는 코딩 속도가 아니라 데이터 축적·라벨 축적·신뢰 축적이라는 세 개의 달력 시간이 결정한다.
